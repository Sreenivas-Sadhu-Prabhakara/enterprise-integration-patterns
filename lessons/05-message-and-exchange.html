The Message & the Exchange: Bodies, Headers, Processors

Meet the two objects every Camel route is built around — and learn to set and read headers. The Exchange is the envelope; the Message inside it is a body plus the headers later EIPs route on.

The idea

Every step in a Camel route is handed an Exchange — the envelope Camel carries down the route, holding an id, exchange properties, and the current message. The Message is what's inside: exchange.getMessage() gives you a body (the payload) plus headers (control metadata that travels alongside the body but is not part of it). You'll do the same job two ways — imperatively with a Processor, and declaratively with the DSL — so the trade-off is concrete.

The scenario

An order arrives on direct:orders as a plain java.util.Map with orderId, country, and amount. Downstream patterns shouldn't have to re-parse the body every time they need to decide, so we lift the routing-relevant fields up into headers once, at the edge: orderId and country imperatively in a Processor, and amount declaratively with setHeader.

The route

MessageAndExchangeRoute.java
from("direct:orders")                                     // (1) orders arrive as a java.util.Map body
    .routeId("exchange")                                  //     always name a route (tracing, metrics, tests)
    .log("IN  exchange ${exchangeId}: the Message body (a Map) = ${body}")

    // (2) IMPERATIVE: a Processor reads exchange.getMessage() and copies fields into headers.
    .process(new OrderHeadersProcessor())

    // (3) DECLARATIVE equivalent: setHeader + the Simple language reading a Map key ${body[key]}.
    .setHeader("amount", simple("${body[amount]}"))

    // (4) Headers now carry the control metadata later EIPs will route on. Log BOTH.
    .log("OUT headers (Processor):   orderId=${header.orderId}, country=${header.country}")
    .log("OUT header  (setHeader):   amount=${header.amount}")

    .to("{{ep.out}}");                                    // (5) placeholder: log: when run, mock: in tests

The imperative half lives in a plain Processor — handed the whole Exchange, reaching into the Message:

OrderHeadersProcessor.java
public void process(Exchange exchange) {
    // (a) The Exchange is the envelope; the Message is what's inside (body + headers).
    Message message = exchange.getMessage();

    // (b) getBody(Type) type-converts for you; the body here already is a Map.
    Map<String, Object> order = message.getBody(Map.class);

    // (c) Lift two body fields into HEADERS — metadata that rides alongside the body.
    message.setHeader("orderId", order.get("orderId"));
    message.setHeader("country", order.get("country"));
}
  1. (1) the route consumes orders from an in-memory direct: channel; each body is a java.util.Map.
  2. (2) .process(...) runs the imperative Processor: (a) get the Message off the Exchange, (b) read its body as a Map, (c) copy orderId and country into headers.
  3. (3) the declarative equivalent — setHeader with the Simple language reading a Map key ${body[amount]} — sets amount with no Java.
  4. (4) both headers now ride with the message; we log them to prove it.
  5. (5) the terminal endpoint is a {{ep.out}} placeholder, so production points it at a real channel while tests point it at a mock. Prefer exchange.getMessage() over the older getIn() — it always returns the message the next step will see.

The test is the spec

MessageAndExchangeTest.java
@Test
void copiesOrderIdAndCountryIntoHeaders() throws Exception {
    out.expectedMessageCount(1);
    // Imperative Processor lifted these two body fields into headers:
    out.expectedHeaderReceived("orderId", "A-2001");
    out.expectedHeaderReceived("country", "IN");
    // Declarative setHeader + Simple ${body[amount]} set this one:
    out.expectedHeaderReceived("amount", 4200);

    Map<String, Object> order = new LinkedHashMap<>();
    order.put("orderId", "A-2001");
    order.put("country", "IN");
    order.put("amount", 4200);
    template.sendBody("direct:orders", order);

    out.assertIsSatisfied();
}

In the test profile ep.out resolves to mock:out, so the test sends a Map order and asserts the message on the mock carries the exact orderId, country, and amount headers — derived from each message, not hardcoded. Run just the tests:

What you learned

  • The Exchange is the envelope Camel carries down a route; the Message inside it is a body + headers pair you reach with exchange.getMessage().
  • Headers carry control metadata that rides alongside the body, so later EIPs route, filter, and aggregate without re-parsing the payload.
  • You can set a header two ways — imperatively in a Processor or declaratively with setHeader(...) + the Simple language — same result, different trade-offs.

Common mistake

Treating headers as durable, exchange-wide state. Headers live on the Message, so they can be dropped or renamed when the message crosses a transport boundary — a jms: hop truncates and mangles header names that don't fit JMS rules, and http: only forwards what you tell it to. For a value that must survive for the whole exchange regardless of endpoint hops, use an exchange property (setProperty(...) / getProperty(...)) instead of a header. Related: always read the current message with exchange.getMessage(), not the older getIn(), or you may act on a stale copy after a step has replaced it.

Production version

Here {{ep.out}} resolves to a log: endpoint and the body is a hand-built java.util.Map; in production it would point at a real channel — jms:, kafka:, or http: — and the payload would typically arrive as JSON you unmarshal into a typed POJO before touching it. Keep ep.out externalized in config (as it already is) so the same route runs against a mock in tests and a broker in prod. Decide deliberately which fields become transport headers: JMS and Kafka need explicit header mapping/serialization, so promote only the small set of routing keys (orderId, country, amount) and keep everything else in the body or in exchange properties. Add observability — Camel's Micrometer metrics and the named routeId("exchange") make each hop traceable — and validate the incoming body so a malformed order fails fast instead of setting garbage headers downstream.

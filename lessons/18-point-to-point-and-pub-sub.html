Point-to-Point vs Publish-Subscribe (Queues & Topics) broker

A queue delivers each message to exactly one consumer; a topic broadcasts a copy to every subscriber. Pick the channel shape by asking: should this be handled once, or heard by everyone?

The idea

Two channels look the same from the sender's chair but behave very differently. A Point-to-Point channel — a queue — competes its consumers: each message is handed to exactly one of them, which is what you want for work that must happen once (ship this order). A Publish-Subscribe channel — a topic — fans out: every subscriber receives its own copy, which is what you want for events others react to independently (an order was placed). In Camel the DSL is identical; only the endpoint prefix changes — jms:queue: versus jms:topic:.

The scenario

ShopFlow moves its critical hand-offs onto a real broker (ActiveMQ Artemis). Confirmed orders go to a fulfillment queue consumed by exactly one warehouse worker — you never want the same order shipped twice. Meanwhile an order-placed event goes to a topic that both the email service and the analytics service subscribe to; each gets its own copy and reacts on its own.

The route

MessagingRoutes.java
public void configure() {
    // ----- Point-to-Point: a queue delivers to exactly ONE consumer -----
    from("direct:fulfillment").routeId("send-to-queue")            // (1) producer side
        .to("jms:queue:fulfillment");                              // (2) send to a QUEUE

    from("jms:queue:fulfillment").routeId("warehouse-worker")      // (3) the one consumer
        .to("{{ep.fulfillment-out}}");

    // ----- Publish-Subscribe: a topic broadcasts to EVERY subscriber -----
    from("direct:order-events").routeId("publish-events")          // (4) publisher side
        .to("jms:topic:order-events");                             // (5) send to a TOPIC

    from("jms:topic:order-events").routeId("email-subscriber")     // (6) subscriber A
        .to("{{ep.email-out}}");

    from("jms:topic:order-events").routeId("analytics-subscriber") // (7) subscriber B
        .to("{{ep.analytics-out}}");
}
  1. (1)(2) the producer drops the message on a jms:queue: — a Point-to-Point channel.
  2. (3) one route consumes that queue. Even if you scaled it to many workers, the broker hands each message to only one of them.
  3. (4)(5) the publisher sends the event to a jms:topic: — a Publish-Subscribe channel.
  4. (6)(7) two independent routes subscribe to the same topic; each receives its own copy. The sinks are {{ep.*}} placeholders so production logs them while tests point them at mocks.

What you learned

  • A queue (Point-to-Point) hands each message to exactly one consumer; a topic (Publish-Subscribe) gives every subscriber its own copy.
  • In Camel the only difference is the endpoint prefix — jms:queue: vs jms:topic: — the rest of the DSL is identical.
  • Broker-backed behavior needs a real broker: make up before spring-boot:run, and Testcontainers under verify for the integration tests.

Related patterns

  • Recipient List — route-level fan-out: the sender picks named destinations, versus a topic where the broker broadcasts to whoever subscribed.
  • Wire Tap — send a copy to a second endpoint without disturbing the main flow; a lightweight, in-route cousin of pub-sub.
  • Request-Reply vs Event Message — a topic almost always carries fire-and-forget events; this compares that style with request-reply.
  • Full EIP Map — see where the Point-to-Point and Publish-Subscribe channels sit among the messaging patterns.

Common mistake

The classic trap with a jms:topic: is assuming subscribers never miss anything. A plain topic subscription is non-durable: if a subscriber route is down — or merely still starting up — when an event is published, that copy is gone for good. A queue holds each message until some consumer takes it, but a topic drops it for anyone not currently listening. If an event truly must survive restarts, declare a durable subscription (a stable clientId plus durableSubscriptionName) or route that consumer through a queue. And don't reach for a topic just because you have several consumers — use one only when each consumer genuinely needs its own copy.

Production version

In production the {{ep.*}} sinks stop being log: endpoints and become real work — an HTTP call to a fulfillment service, a database write, or another broker hop — while the broker URL, user and password move out of application.yaml into environment variables or a secrets manager. For durability you back the queue with persistent storage and add a Dead Letter queue with redelivery, and because a plain JMS topic is non-durable you register durable subscriptions so the email and analytics services don't lose events across a restart. To scale the warehouse worker you add competing consumers (concurrentConsumers) behind the same queue — the broker still hands each order to exactly one — and you wire up metrics and queue-depth alerts so a backed-up channel is visible. For very high event volume teams often swap JMS for kafka:: a single consumer group gives Point-to-Point, distinct group ids give Publish-Subscribe, and partitioned, disk-retained topics add the replay and parallelism JMS topics don't offer.

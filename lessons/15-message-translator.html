Message Translator

Convert a message from one format to another so two systems that speak different "languages" can integrate. Reach for it whenever an inbound payload doesn't match the shape your system expects.

The idea

Two systems rarely agree on wording. A Message Translator is the interpreter between them: it sits in the middle and reshapes each message so the receiver understands it. In Camel the work splits in two. A Data Format — the reusable codec you plug into unmarshal/marshal — moves you between bytes/String and a POJO. The translation proper — renaming fields, converting units — lives in a small bean invoked with .transform().method(...). Keep the mapping in a bean and the route stays readable and the mapping stays unit-testable on its own.

The scenario

A partner posts orders in a terse legacy JSON shape — abbreviated keys and money as an integer number of cents: { "id": "A-1001", "cust": "Acme Corp", "amt": 123499, "cur": "USD" }. Our system speaks a clean canonical shape with readable names and money in whole currency units: { "orderId": "A-1001", "customer": "Acme Corp", "amount": 1234.99, "currency": "USD" }. The route does it in three moves — unmarshal to a LegacyOrder, transform to a canonical Order, then marshal back to JSON.

The route

MessageTranslatorRoute.java
from("direct:orders")                                         // (1) legacy JSON arrives as a String
    .routeId("translator")                                    //     always name a route
    .log("Translating legacy order: ${body}")
    .unmarshal().json(JsonLibrary.Jackson, LegacyOrder.class)  // (2) Data Format: JSON String -> LegacyOrder
    .transform().method(OrderMapper.class, "toCanonical")     // (3) Message Translator: LegacyOrder -> Order
    .marshal().json(JsonLibrary.Jackson)                      // (4) Data Format: Order -> canonical JSON
    .log("Emitting canonical order: ${body}")
    .to("{{ep.out}}");                                        // (5) downstream (log: in run, mock: in test)
  1. (1) the route consumes a legacy JSON String from an in-memory direct: channel.
  2. (2) unmarshal().json(JsonLibrary.Jackson, LegacyOrder.class) uses the Jackson Data Format to bind the JSON onto a LegacyOrder POJO — no annotations needed because its field names match the legacy keys.
  3. (3) .transform().method(OrderMapper.class, "toCanonical") is the Message Translator proper: the bean renames fields (id → orderId, cust → customer, cur → currency) and converts cents to whole currency units.
  4. (4) marshal().json(JsonLibrary.Jackson) runs the same Data Format the other way, serialising the canonical Order back to JSON.
  5. (5) {{ep.out}} is a property placeholder — production points it at a real direct:/jms: endpoint while tests point it at a mock: endpoint.

The test is the spec

MessageTranslatorTest.java
@Test
void legacyJsonIsTranslatedToCanonicalJson() throws Exception {
    out.expectedMessageCount(1);

    // LEGACY_JSON = { "id":"A-1001","cust":"Acme Corp","amt":123499,"cur":"USD" }
    template.sendBody("direct:orders", LEGACY_JSON);

    out.assertIsSatisfied();

    // The emitted body is canonical JSON. Parse it and assert on meaning, not byte order.
    String canonical = out.getExchanges().get(0).getMessage().getBody(String.class);
    JsonNode node = JSON.readTree(canonical);

    assertEquals("Acme Corp", node.get("customer").asText());   // renamed: cust -> customer
    assertEquals("A-1001", node.get("orderId").asText());       // renamed: id   -> orderId
    // converted: 123499 cents -> 1234.99 currency units
    assertEquals(0, new BigDecimal("1234.99").compareTo(node.get("amount").decimalValue()));

    // The legacy keys must be gone from the canonical output.
    assertTrue(node.get("cust") == null, "legacy key 'cust' should not survive translation");
    assertTrue(node.get("amt") == null, "legacy key 'amt' should not survive translation");
}

In the test profile ep.out resolves to mock:out, so the test parses the JSON that lands there and asserts on the renamed field (cust → customer) and the converted amount (123499 cents → 1234.99) — and that the legacy keys are gone. Run just the tests:

What you learned

  • A Message Translator reshapes a message so systems that speak different formats can integrate.
  • A Data Format (Jackson) is the reusable codec for marshal/unmarshal between JSON and POJOs; a mapping bean does the renaming and unit conversion a codec can't.
  • Property-placeholder endpoints ({{ep.out}}) let the same route be production-real and test-assertable.

Common mistake

Expecting the Data Format to do the translation. Jackson's unmarshal/marshal only binds fields whose names already match — it will never rename cust to customer or turn 123499 cents into 1234.99. If you drop the OrderMapper and marshal the LegacyOrder straight through, the terse legacy keys leak downstream unchanged. Keep the codec and the .transform().method(...) mapping as two separate steps, and do the money math with BigDecimal.movePointLeft(2) — never a double, which quietly loses cents to floating-point rounding.

Production version

In this lesson the legacy JSON arrives on an in-memory direct: channel from a demo timer, and {{ep.out}} resolves to log:out. In production the payload would land on a real transport — kafka:orders-legacy or an http:/jms: consumer — and the out endpoint would resolve to a durable jms:queue:orders so translated orders survive restarts and can be consumed by an independently scaled service. Externalize both endpoint URIs and any Jackson tuning (date format, fail-on-unknown-properties) into application.yaml per environment. The OrderMapper bean is stateless, so Camel can run many exchanges through it concurrently without locking — keep it that way. Finally, wrap the route in a Dead Letter Channel so a malformed legacy payload Jackson can't unmarshal goes to an error queue for inspection instead of stalling the flow, and log the orderId on each translation for traceability.

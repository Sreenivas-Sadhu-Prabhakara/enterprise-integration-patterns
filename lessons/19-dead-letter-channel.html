Dead Letter Channel & Redelivery

When a step fails, don't drop the message — retry it a few times, then move the exhausted exchange to a dedicated dead letter channel. Reach for it whenever the downstream might be flaky or the payload might be bad, and silently losing the message is unacceptable.

The idea

A failed delivery has two bad endings: throw the message away, or crash the whole route. A Dead Letter Channel gives it a third: retry a bounded number of times, and if it still won't go through, divert it to a side channel where a human or a replay job can deal with it later. Two knobs do the work — the redelivery policy (maximumRedeliveries, redeliveryDelay, optionally backOffMultiplier to grow the delay) and the deadLetterChannel(endpoint) that receives the give-up message. Because the channel handles the exhausted exchange, the caller never sees the exception — unlike Camel's default handler, which rethrows.

The scenario

ShopFlow charges each order through a payment gateway (PaymentBean). A good card (failPayment=false) clears on the first try and flows on to the OK channel. A permanently declined card (failPayment=true) throws every time: it is retried 3×, 20 ms apart, and once exhausted it lands on the Dead Letter Channel instead of being lost.

The route

DeadLetterChannelRoute.java
from("direct:orders")                                    // (1) the channel orders arrive on
    .routeId("dlq")                                      //     always name a route
    .errorHandler(deadLetterChannel("{{ep.dead}}")       // (2) the Dead Letter Channel itself
        .maximumRedeliveries(3)                          // (3) retry up to 3 times before giving up
        .redeliveryDelay(20))                            // (4) wait 20 ms between attempts (tiny = fast tests)
    .log("Charging order ${body.orderId} (amount ${body.amount})")
    .bean("paymentBean", "charge")                       // (5) may throw PaymentDeclinedException
    .log("Order ${body.orderId} paid OK")
    .to("{{ep.ok}}");                                    // (6) success channel — never reached on failure
  1. (1) the route consumes orders from an in-memory direct: channel.
  2. (2) errorHandler(deadLetterChannel(...)) scopes the policy to this whole route — it catches any exception. For per-type handling you'd add onException(...) clauses; the most specific one wins and this stays the fallback.
  3. (3)(4) the redelivery policy: maximumRedeliveries(3) gives up after three retries, redeliveryDelay(20) waits 20 ms between attempts (keeping the test sub-second).
  4. (5) .bean("paymentBean", "charge") is the fallible step; a declined card throws PaymentDeclinedException every call.
  5. (6) on success the exchange continues to {{ep.ok}}; on exhaustion it goes to {{ep.dead}}. Both are {{ep.*}} placeholders, so production points them at real channels while tests point them at mocks.

Run it

terminal
./mvnw -pl patterns/19-dead-letter-channel spring-boot:run

A demo feeder injects a good order and a declined order in alternation every 5s — watch the good ones log paid OK, and the declined ones make three retry attempts before landing on the log:dead endpoint with their CamelRedeliveryExhausted header showing. Behind a firewall? Add -P upstream.

The test is the spec

DeadLetterChannelTest.java
@Test
void failingOrderLandsInDeadLetterAfterRetries() throws Exception {
    ok.expectedMessageCount(0);
    dead.expectedMessageCount(1);

    template.sendBody("direct:orders", new Order("A-1002", 500, true));

    MockEndpoint.assertIsSatisfied(context);
    // 1 original attempt + 3 redeliveries (maximumRedeliveries(3)) = 4 calls into the gateway.
    assertEquals(4, paymentBean.getAttempts());
}

In the test profile the terminal endpoints resolve to mock:ok and mock:dead, so the test asserts exactly where each order ends up — no broker required. The declined order reaches mock:dead with zero on mock:ok, and the bean counter proves the retries happened: four gateway calls (1 original + 3 redeliveries). The sibling test sends a good order and asserts one call and zero dead letters. Run just the tests:

terminal
./mvnw -pl patterns/19-dead-letter-channel test
Try it Add .backOffMultiplier(2) to the redelivery policy and watch the delay grow 20ms → 40ms → 80ms. Then add an onException(PaymentDeclinedException.class) clause with maximumRedeliveries(0) so declined cards go straight to dead, and assert getAttempts() is exactly 1.

What you learned

  • A Dead Letter Channel keeps a message that can't be delivered instead of dropping it — retry a bounded number of times, then divert.
  • A redelivery policy (maximumRedeliveries / redeliveryDelay / backOffMultiplier) controls how hard, and how gently, to retry.
  • Because the channel handles the exhausted exchange, the caller never sees the exception — the message is safe on the dead endpoint, not lost.

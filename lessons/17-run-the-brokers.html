Run the Brokers

Stand up real message brokers — ActiveMQ Artemis and Kafka — so the next lessons talk to real infrastructure instead of in-memory channels. This is the moment the course crosses from logic into operations.

Reality check Lessons 1–16 used in-memory channels (direct:, seda:) so you could focus on the pattern. Real endpoints cross a network: they add latency, they drop connections, and they can deliver the same message twice. A broker is where those realities begin — and it's exactly why Dead Letter Channels and Idempotent Consumers exist.

Why a broker at all?

An in-memory seda: queue vanishes when the JVM stops. A broker is a separate process that holds messages durably and lets independent services hand work off without knowing about each other. That buys you three things you can't get in-memory: durability (messages survive a crash), decoupling (producer and consumer run and scale separately), and cross-process delivery (real services, real network).

  • ActiveMQ Artemis — a JMS broker; the home of the queue (point-to-point) and topic (publish-subscribe) lessons.
  • Apache Kafka — a distributed log for high-throughput event streaming; the same pub/sub idea with consumer groups.

The scenario

ShopFlow is ready to move its critical hand-offs off in-memory channels and onto a broker — confirmed orders to a durable fulfillment queue, order events to a topic. First, bring the infrastructure up.

Start the brokers

The code repo ships an infra/docker-compose.yml with Artemis, Kafka, and a Kafka UI. One command brings them up:

terminal
make up
# equivalently:
docker compose -f infra/docker-compose.yml up -d
expected output
[+] Running 4/4
 ✔ Container eip-artemis    Healthy
 ✔ Container eip-kafka      Started
 ✔ Container eip-kafka-ui   Started
 ✔ Network camel-eip_default Created

What's now running

ServicePortUse
ActiveMQ Artemis (JMS)61616spring.artemis.broker-url=tcp://localhost:61616
Artemis web console8161http://localhost:8161 (camel / camel)
Kafka9092camel.component.kafka.brokers=localhost:9092
Kafka UI8080http://localhost:8080 — watch messages land

Verify & tear down

terminal
docker compose -f infra/docker-compose.yml ps    # both brokers up?
make logs                                         # tail broker logs
make down                                          # stop and remove everything
No Docker? You don't need this running to pass the tests. The messaging lesson's tests use Testcontainers, which start a throwaway broker automatically under ./mvnw verify. make up is only for running the app by hand and watching messages flow in the consoles.

What you learned

  • A broker is a separate, durable process — the opposite of an in-memory seda: queue.
  • make up starts Artemis + Kafka + a Kafka UI from infra/docker-compose.yml.
  • Crossing a broker introduces the real-world failures the reliability lessons exist to handle.

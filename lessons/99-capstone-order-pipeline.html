Capstone: the End-to-End ShopFlow Pipeline

Assemble everything you've learned into one coherent order-processing pipeline. The capstone shows how eight patterns compose — and why their order matters: de-duplicate before side effects, wrap payment in a dead-letter channel.

Reality check This capstone puts eight patterns in one route so you can see them compose. In production you would not ship it as a monolith — you'd split it across services (bounded contexts), connected by real brokers, each deployed and scaled on its own. ShopFlow is the teaching spine; the patterns are what you keep.

The idea

No single pattern ships an order. Real integrations are compositions: each stage's output is the next stage's input, and the value is in how they chain. This capstone wires eight EIPs from the course into one route — a Message Translator, Message Filter, Idempotent Consumer, Wire Tap, Content-Based Router, Request-Reply, Multicast, and Event Message — with a Dead Letter Channel catching failures. Every hop is in-memory (direct:/mock:), so it runs and tests green with no broker.

The scenario

A JSON order enters ShopFlow and flows through eight patterns: translate to an Order → filter out test and zero-value orders → de-duplicate by orderId → send an audit copy → route by country → authorize payment (waiting for the reply) → fan out to warehouse + invoicing → publish an order-placed event. A declined payment is retried once, then parked in a dead-letter channel instead of being lost.

The route

ShopFlowRoute.java
// (0) Dead Letter Channel: a declined payment is retried once, then parked (not lost).
errorHandler(deadLetterChannel("{{ep.dead}}").maximumRedeliveries(1).redeliveryDelay(10));

// (1) Message Translator — JSON text to a canonical Order POJO.
from("direct:orders").routeId("intake")
    .unmarshal().json(JsonLibrary.Jackson, Order.class)
    .to("direct:validate");

// (2) Message Filter — QA/test and zero-value orders never continue.
from("direct:validate").routeId("validate")
    .filter(simple("${body.totalAmount} > 0 && ${body.testFlag} == false"))
        .to("direct:dedupe")
    .end();

// (3) Idempotent Consumer — each orderId is processed exactly once.
from("direct:dedupe").routeId("dedupe")
    .idempotentConsumer(simple("${body.orderId}"),
            MemoryIdempotentRepository.memoryIdempotentRepository())
        .to("direct:process")
    .end();

// (4) Wire Tap (audit) + (5) Content-Based Router (region).
from("direct:process").routeId("process")
    .wireTap("{{ep.audit}}")
    .choice()
        .when(simple("${body.country} == 'IN'")).setHeader("region", constant("domestic"))
        .otherwise().setHeader("region", constant("international"))
    .end()
    .to("direct:pay");

// (6) Request-Reply — call the payment gateway (InOut) and keep the order as the body.
from("direct:pay").routeId("pay")
    .setHeader("orderBackup", body())
    .to(ExchangePattern.InOut, "direct:authorize")   // body becomes the auth code (the reply)
    .setHeader("authCode", body())
    .setBody(header("orderBackup"))                   // restore the Order as the body
    .to("direct:fulfil");

from("direct:authorize").routeId("payment-gateway")
    .process(e -> {
        Order o = e.getMessage().getBody(Order.class);
        if (o.isFraudulent()) {
            throw new PaymentDeclinedException("payment declined for " + o.getOrderId());
        }
        e.getMessage().setBody("AUTH-" + o.getOrderId());
    });

// (7) Multicast fan-out + (8) Event Message (fire-and-forget).
from("direct:fulfil").routeId("fulfil")
    .log("Fulfilling ${body.orderId} (auth ${header.authCode}) region=${header.region}")
    .multicast().to("{{ep.warehouse}}", "{{ep.invoicing}}").end()
    .setExchangePattern(ExchangePattern.InOnly).to("{{ep.event}}");
  1. (0) the Dead Letter Channel wraps the whole context: a decline is retried once, then parked at {{ep.dead}} — never silently dropped.
  2. (1) the Message Translator unmarshals inbound JSON text into a canonical Order POJO the rest of the flow can reason about.
  3. (2) the Message Filter lets only real, payable orders through; QA/test flags and zero-value totals stop here.
  4. (3) the Idempotent Consumer, keyed on ${body.orderId}, guarantees each order is processed exactly once — before any side effect fires.
  5. (4)(5) the Wire Tap forks an audit copy while the main flow continues, then the Content-Based Router tags a region header by country.
  6. (6) Request-Reply is an InOut call to the payment gateway; the reply (an auth code) comes back, so we stash the order first and restore it as the body afterwards.
  7. (7)(8) Multicast fans the authorized order to warehouse + invoicing, then an InOnly Event Message publishes order-placed fire-and-forget. The {{ep.*}} placeholders are real channels in production and mocks under test.

The test is the spec

ShopFlowPipelineTest.java
@Test
void validDomesticOrderFlowsAllTheWay() throws Exception {
    warehouse.expectedMessageCount(1);
    invoicing.expectedMessageCount(1);
    event.expectedMessageCount(1);
    audit.expectedMessageCount(1);
    dead.expectedMessageCount(0);

    template.sendBody("direct:orders", order("OK-IN-1", "IN", 999, false, false));

    MockEndpoint.assertIsSatisfied(context);
}

The order(...) helper builds the JSON string, and in the test profile every {{ep.*}} resolves to a mock: endpoint — so this one assertion proves the full happy path: an audit copy is tapped, warehouse and invoicing both fire, an event is emitted, and nothing lands in the dead-letter channel. The three sibling tests prove test/zero orders are filtered, a duplicate orderId is processed once, and a fraudulent order's decline lands in mock:dead. Run just the tests:

What you learned

  • Real integrations are compositions: eight EIPs chain into one pipeline where each stage's output feeds the next.
  • Order matters — de-duplicate before any side effect, tap for audit before payment, and wrap payment in a Dead Letter Channel so declines are parked, not lost.
  • Request-Reply (InOut) waits for the auth reply; an Event Message (InOnly) fires order-placed and moves on — the same route uses both deliberately.

Common mistake

In a composed pipeline the bug is almost never in one pattern — it is in the order. Placing the Wire Tap audit or the payment call before the Idempotent Consumer means a re-sent order gets audited and charged twice; always de-duplicate before any side effect. A second trap is durability: MemoryIdempotentRepository lives inside a single JVM, so it forgets every orderId on restart and is not shared across instances — fine for this in-memory demo, wrong for real traffic.

Production version

In production the {{ep.*}} placeholders stop being log:/mock: and become real transports: jms: or kafka: queues for warehouse, invoicing, and the order-placed event, an http: call (with timeouts and a circuit breaker) to the real payment gateway, and a genuine broker DLQ for {{ep.dead}}. Swap the in-memory idempotent repository for a JdbcIdempotentRepository or Redis/Infinispan store so de-duplication survives restarts and works across a scaled-out cluster. Externalize the endpoint URIs, credentials, and redelivery policy into config/secrets rather than hard-coding them, and add Micrometer metrics plus distributed tracing so you can follow one orderId across every hop.

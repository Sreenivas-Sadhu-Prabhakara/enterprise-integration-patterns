Testing Camel Routes: MockEndpoint & AdviceWith

Prove a route does exactly what you claim — in memory, with no broker. This module is not a new EIP: it is the Camel test toolkit itself — MockEndpoint, ProducerTemplate, NotifyBuilder, and AdviceWith.

The idea

A route is a claim: "send this in, that comes out over there." A test turns the claim into an executable spec. Camel ships a small toolkit for it that runs entirely in memory: a MockEndpoint records every message so you can assert what arrived and where; a ProducerTemplate feeds messages in; a NotifyBuilder waits for asynchronous routes to settle; and AdviceWith rewrites a route at test time without touching production code. No broker, no network — the tests are fast and deterministic.

The scenario

ShopFlow needs its routes covered by fast tests. We put two deliberately trivial routes under test. The sample route is InOnly: bodies containing VIP go to ep.a (priority), everything else goes to ep.b (standard). The echo route is InOut: it transforms the body to echo:<body> and returns it. The branch targets are property placeholders ({{ep.a}}, {{ep.b}}) so tests resolve them to mock:priority / mock:standard and assert exactly which branch a message took.

The routes

SampleRoute.java
from("direct:orders")
    .routeId("sample")                                 // (1) ALWAYS name a route — tests advise it by id
    .log("sample received: ${body}")
    .choice()                                          // (2) an InOnly content-based split
        .when(simple("${body} contains 'VIP'"))        // (3) the branch a MockEndpoint asserts on
            .to("{{ep.a}}")                            //     priority lane  (mock:priority in tests)
        .otherwise()                                   // (4) everything else
            .to("{{ep.b}}")                            //     standard lane  (mock:standard in tests)
    .end();

from("direct:echo")                                    // (5) an InOut (request/reply) route
    .routeId("echo")
    .transform(simple("echo:${body}"));                //     transform() sets the reply body
  1. (1) the route consumes from an in-memory direct: channel; naming it lets AdviceWith target it by id at test time.
  2. (2) choice() opens a content-based split; the two lanes are {{ep.*}} placeholders so production points them at real endpoints while tests point them at mocks.
  3. (3)(4) the Simple predicate ${body} contains 'VIP' sends VIP orders to the priority lane; otherwise() catches the rest.
  4. (5) a second, InOut route transforms the body to echo:<body> and returns it — the target for a requestBody request/reply test.

Run it

terminal
./mvnw -pl patterns/06-testing-camel-routes spring-boot:run

A demo feeder injects a rotating body every 3s (VIP order #N / standard order #N), so you can watch the sample route log each message and land it on the matching branch. Behind a firewall? Add -P upstream for plain Apache Camel.

The test is the spec

Each method demonstrates one technique. First, a MockEndpoint declares up-front what should arrive — expectedBodiesReceived also pins the exact count — then assertIsSatisfied verifies it:

SampleRouteTest.java
@Test
void vipBodyReachesPriorityWithExpectedBody() throws Exception {
    priority.expectedBodiesReceived("VIP order for alice");
    standard.expectedMessageCount(0);

    template.sendBody("direct:orders", "VIP order for alice");

    MockEndpoint.assertIsSatisfied(context);
}

In the test profile {{ep.a}}/{{ep.b}} resolve to mock:priority and mock:standard, so the test proves the VIP body took the priority branch and nowhere else. Next, AdviceWith rewrites the route at test time — weaving mock:tap onto the end so we can assert on the output no matter which branch fired. Because it needs a not-yet-started context (@UseAdviceWith), it lives in its own class:

AdviceWithSampleRouteTest.java
@Test
void weaveAddLastLetsUsTapTheRouteOutput() throws Exception {
    // (1) Weave an extra step onto the END of the "sample" route — runs after whichever branch fired.
    AdviceWith.adviceWith(context, "sample", rb -> rb.weaveAddLast().to("mock:tap"));

    // (2) With @UseAdviceWith the context is NOT auto-started — start it now that weaving is done.
    context.start();

    // (3) Exercise and assert. A non-VIP body takes the otherwise branch, then still hits mock:tap.
    tap.expectedBodiesReceived("weekend delivery");

    template.sendBody("direct:orders", "weekend delivery");

    tap.assertIsSatisfied();
}

Run just the tests — two classes, five methods, no broker:

terminal
./mvnw -pl patterns/06-testing-camel-routes test
Try it Weave a mock:tap onto the echo route with AdviceWith and assert that echo:hello flows out — or add a NotifyBuilder that waits for one exchange from direct:echo to complete.

What you learned

  • A MockEndpoint records messages so you assert exactly which branch a message took — the test becomes the route's spec.
  • sendBody drives InOnly routes, requestBody drives InOut and returns the reply, and NotifyBuilder waits for asynchronous completion.
  • AdviceWith rewrites a route at test time (weaveAddLast().to("mock:tap")) without touching production code — but it needs @UseAdviceWith and a manual context.start().

When a Route Throws: Error Handling Basics

Control what happens when a step in a route throws. Learn the three everyday tools — the default error handler, onException + handled(true), and doTry/doCatch — plus a first look at redelivery and a dead-letter destination.

The idea

By default Camel wraps every route in a default error handler: if a step throws and nothing catches it, the exception is logged and propagated back to the caller, the exchange is marked failed, and the rest of the route is skipped. Nothing is retried, nothing is diverted — it just blows up. Three deliberate strategies replace that: onException(...).handled(true) catches, logs and diverts (and hides the failure from the caller); errorHandler(deadLetterChannel(...)) retries a few times and then parks the message for ops to replay; and doTry()/doCatch() handles a failure inline, exactly like Java's try/catch.

The scenario

ShopFlow charges a card in PaymentBean. Orders flagged failPayment=true make the bean throw a RuntimeException — a declined card, a downstream 500, a bug. We handle that failure three ways, one route each: error-handling (onException) diverts to ep.error, dlc-demo (deadLetterChannel) retries then parks on ep.dead, and try-catch-demo (doTry/doCatch) catches inline to ep.caught. This page walks the first, most common one.

The route

ErrorHandlingRoute.java
onException(RuntimeException.class)                       // (1) catch this exception thrown anywhere below
    .handled(true)                                       // (2) "I've dealt with it" — do NOT rethrow to the caller
    .log("Payment FAILED for ${body.orderId}: ${exception.message} -> error channel")
    .to("{{ep.error}}");                                 // (3) divert the bad message to the error destination

from("direct:orders")                                    // (4) orders arrive on this channel
    .routeId("error-handling")                           //     always name a route (tracing, metrics, tests)
    .log("Charging order ${body.orderId}")
    .bean("paymentBean", "process")                      // (5) may throw RuntimeException -> caught by (1)
    .log("Payment OK for ${body.orderId}")
    .to("{{ep.ok}}");                                    // (6) happy path: only reached if (5) did NOT throw
  1. (1) onException registers a route-wide catch that applies to every route in this RouteBuilder; it fires for the listed exception (and its subclasses) thrown anywhere downstream.
  2. (2) handled(true) tells Camel the failure is dealt with, so the exception is not rethrown — the caller sees success, not a stack trace.
  3. (3) the failed message is diverted to {{ep.error}} instead of aborting the exchange.
  4. (4)(5) the happy path consumes an order, logs, then calls PaymentBean.process, which throws for any flagged order.
  5. (6) {{ep.ok}} is reached only when the bean did not throw; on a throw, control jumps to the onException block. Both targets are {{ep.*}} placeholders — log: in the app, mock: in tests.

A demo feeder injects an order every 3s, alternating good and failing. Watch Charging order A-1001 land on log:ok, or Payment FAILED … -> error channel land on log:error — proof that a thrown step no longer aborts the app. Behind a firewall? Add -P upstream.

The test is the spec

ErrorHandlingTest.java
@Test
void failingOrderIsHandledToError() throws Exception {
    ok.expectedMessageCount(0);
    error.expectedMessageCount(1);

    // handled(true) means this does NOT throw back to us even though the bean threw.
    template.sendBody("direct:orders", new Order("A-1002", 500, true));

    MockEndpoint.assertIsSatisfied(context);
}

In the test profile every endpoint resolves to a mock: (see src/test/resources/application.yaml), so each test asserts exactly where a message ended up — no broker. Because the bean threw but handled(true) is set, sendBody returns normally: the failure lands on mock:error and mock:ok stays empty. Run just the tests:

Try it Flip handled(true) to handled(false) in ErrorHandlingRoute and rerun the test — watch the exception get rethrown to the caller. Then drop maximumRedeliveries from 2 to 0 in DeadLetterChannelRoute and assert the failing charge still lands on mock:dead, now with no retries.

What you learned

  • By default a thrown step aborts the route and propagates the exception back to the caller; error handling replaces that with something deliberate.
  • onException + handled(true) catches, logs and diverts — and hides the failure from the caller; a dead-letter channel first retries then parks; doTry/doCatch handles inline like Java's try/catch.
  • Property-placeholder endpoints (log: in the app, mock: in tests) make every branch assertable without a broker.

Common mistake

Diverting a failure with onException(...).to("…error") but forgetting handled(true). The message does reach the error channel, yet Camel still marks the exchange failed and rethrows to the caller — so sendBody throws and nothing was actually "handled". Set handled(true) only when you have genuinely dealt with the failure; slapping it on everything silently swallows real bugs. Match the flag to intent, and keep onException scoped to the specific exception you expect rather than a blanket Exception.class.

Production version

The teaching routes read from direct: and write to log:; in production orders arrive on a real consumer (jms:, kafka: or http:) and the dead-letter target is a durable queue such as jms:queue:orders.DLQ, not a log line — so parked messages survive a restart and ops can inspect and replay them. Externalize the redelivery policy (maximumRedeliveries, an exponential redeliveryDelay with backOffMultiplier) into config instead of hard-coding it, and pair retries with an idempotent consumer so a redelivered charge is never applied twice. Add route-level metrics and tracing (Micrometer, exchange counters) and alert on DLQ depth, so a growing pile of dead letters pages someone rather than scrolling past in a console.

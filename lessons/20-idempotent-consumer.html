Idempotent Consumer: Process Each Order Once

Process each message exactly once by dropping duplicates. Reach for it whenever the same logical message can arrive twice — a broker redelivery, a replayed batch, a double-tapped "Pay" — and repeating the work would be harmful.

The idea

Picture a bouncer with a guest list, ticking off each name as it arrives. Show up a second time and you're turned away — you're already on the list. An Idempotent Consumer does the same for messages: it derives a stable idempotency key from each one and remembers the keys it has already seen. In Camel that's the idempotentConsumer(key, repository) block — the first message carrying a key runs the guarded steps; every later copy of that key is silently dropped. The repository is the guest list; the key must be identical across every copy of the same logical message.

The scenario

ShopFlow receives orders on direct:orders. Every order carries a stable idempotency key in its orderId header. The first A-1001 is new, so it is processed and reaches ep.out. When the broker redelivers the same A-1001, its key is already seen, so it is dropped. A distinct order A-1002 is a genuinely new key and passes through.

The route

IdempotentConsumerRoute.java
from("direct:orders")                                            // (1) the channel orders arrive on
    .routeId("idempotent")                                       //     always name a route (tracing, metrics, tests)
    .idempotentConsumer(                                          // (2) the Idempotent Consumer itself
            header("orderId"),                                    // (3) the idempotency key
            MemoryIdempotentRepository.memoryIdempotentRepository()) // (4) in-memory seen-keys store (swap for JDBC in prod)
        .log("Processing order ${header.orderId} (amount=${body.amount})") // only first-seen keys reach here
        .to("{{ep.out}}")                                        // (5) the exactly-once destination
    .end();                                                      //     close the idempotent block
  1. (1) the route consumes orders from an in-memory direct: channel.
  2. (2) idempotentConsumer(...) opens the guarded block; only the first message for a given key runs the steps inside it.
  3. (3) the key is header("orderId") — a stable, business-meaningful value that is identical across every duplicate delivery of the same order.
  4. (4) MemoryIdempotentRepository is the seen-keys store: fine for one JVM and for tests, but it forgets on restart. Swap in a JDBC / Redis / Infinispan repository for restart-safe, cluster-wide dedup — the DSL is unchanged.
  5. (5) .to("{{ep.out}}") is the exactly-once destination. The {{ep.out}} placeholder points at a real channel in production and at a mock in tests.

Run it

terminal
./mvnw -pl patterns/20-idempotent-consumer spring-boot:run

A demo feeder emits each order twice every few seconds — watch the first copy log as Processing order A-1000 … and its immediate duplicate get silently dropped. Behind a firewall? Add -P upstream.

The test is the spec

IdempotentConsumerTest.java
@Test
void duplicatesAreDroppedButDistinctIdsPassThrough() throws Exception {
    // Same orderId sent twice — the second is a duplicate and must be silently dropped.
    out.expectedMessageCount(1);

    template.sendBodyAndHeader("direct:orders", new Order("A-1001", 999), "orderId", "A-1001");
    template.sendBodyAndHeader("direct:orders", new Order("A-1001", 999), "orderId", "A-1001"); // duplicate

    out.assertIsSatisfied();   // fired EXACTLY once

    // A second, DISTINCT orderId is a genuinely new order — it passes, bringing the total to 2.
    out.expectedMessageCount(2);

    template.sendBodyAndHeader("direct:orders", new Order("A-1002", 500), "orderId", "A-1002");

    out.assertIsSatisfied();   // now 2 total
}

In the test profile {{ep.out}} resolves to mock:out, so the test asserts exactly how many messages survive deduplication: the twice-sent A-1001 fires once, and the distinct A-1002 brings the total to two. Run just the tests:

terminal
./mvnw -pl patterns/20-idempotent-consumer test
Try it Send a third delivery of A-1001 after A-1002 and assert mock:out stays at 2 — proving the key is remembered for the whole run, not just against the previous message.

What you learned

  • An Idempotent Consumer processes each message exactly once by remembering an idempotency key and dropping duplicate deliveries.
  • idempotentConsumer(key, repository) is the Camel DSL — the key must be stable across every copy of the same logical message.
  • The repository is swappable: in-memory for one JVM and tests, JDBC / Redis / Infinispan for restart-safe, cluster-wide dedup.

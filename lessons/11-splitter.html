Splitter: Break a Bulk Order into Line Items

Transmit one composite message as a sequence of smaller messages. Reach for it whenever a single payload really carries "a list of things" and each element must be handled on its own.

The idea

Picture a bulk order arriving in one envelope with a packing list inside. A Splitter opens it and puts each line item onto the belt as its own message. In Camel that's split(expression) — it evaluates an expression that yields a collection and emits one exchange per element. Camel stamps every sub-message with CamelSplitIndex (its 0-based position) and, unless you add .streaming(), CamelSplitSize (the total), so a later step can track progress or re-assemble the pieces.

The scenario

ShopFlow receives a bulk Order that bundles several line items — Order{ orderId: A-1001, items: [SKU-1, SKU-2, SKU-3] } should become the three messages SKU-1, then SKU-2, then SKU-3. We fan the list out with split(simple("${body.items}")), sending each item to a {{ep.item}} placeholder — a log: endpoint when you run the app, a mock: endpoint under test.

The route

SplitterRoute.java
from("direct:orders")                                          // (1) a bulk order arrives as ONE message
    .routeId("splitter")
    .log("Splitting order ${body.orderId}")
    .split(simple("${body.items}"))                            // (2) the Splitter: fan the list out
        // .streaming()                                        //     enable for huge inputs: constant
        //                                                     //     memory, but CamelSplitSize is
        //                                                     //     unknown until the final element
        .log("item ${header.CamelSplitIndex} of ${header.CamelSplitSize}: ${body}") // (3) per-item
        .to("{{ep.item}}")                                     // (4) each item continues as its own message
    .end();                                                    // (5) split ends; the original exchange resumes
  1. (1) the route consumes a bulk order from an in-memory direct: channel — one message carrying a list.
  2. (2) split(simple("${body.items}")) is the Splitter: it evaluates the expression to a collection and emits one exchange per element.
  3. (3) inside the split block each sub-message is logged with the CamelSplitIndex and CamelSplitSize headers Camel stamps automatically.
  4. (4) each item flows on to the {{ep.item}} placeholder so production points it at a real handler while tests point it at a mock.
  5. (5) .end() closes the split and the original exchange resumes. For very large lists add .streaming() to iterate lazily — at the cost of an unknown CamelSplitSize until the last element.

The test is the spec

SplitterTest.java
@Test
void splitsBulkOrderIntoOneMessagePerLineItem() throws Exception {
    // ONE order in with THREE items -> THREE messages out, one per item, in order.
    item.expectedMessageCount(3);
    item.expectedBodiesReceived("SKU-1", "SKU-2", "SKU-3");

    template.sendBody("direct:orders",
            new Order("A-1001", List.of("SKU-1", "SKU-2", "SKU-3")));

    MockEndpoint.assertIsSatisfied(context);
}

In the test profile {{ep.item}} resolves to mock:item, so the test asserts exactly three messages fan out with the right bodies in order. A second test proves every sub-message carries the right CamelSplitIndex (0, 1, 2) and CamelSplitSize (3). Run just the tests:

What you learned

  • A Splitter turns one composite message into a sequence of smaller messages, one per element.
  • split(expression) is the Camel DSL — and Camel stamps each sub-message with CamelSplitIndex and (unless streaming) CamelSplitSize.
  • A property-placeholder per-item endpoint keeps the same route production-real and test-assertable; .streaming() trades a known size for constant memory on huge inputs.

Related patterns

  • 12 · Aggregator — the natural inverse: it recombines the sub-messages a Splitter emits, and can use CamelSplitSize as its completion signal to know when every piece has arrived.
  • 13 · Recipient List — the other fan-out: a Splitter emits one message per element of the body, while a Recipient List emits one per destination in a computed list.
  • 08 · Pipes & Filters — each split sub-message flows on as its own exchange, so downstream it typically enters its own pipeline of processing steps.

Common mistake

Adding .streaming() and then still depending on CamelSplitSize downstream. When you stream, Camel iterates the collection lazily and does not know the total up front, so CamelSplitSize stays -1 on every sub-message except the last — any "item X of N" logic or a size-based completion in a later Aggregator silently breaks. Only enable streaming when the input is genuinely too large to materialise, and switch downstream logic to CamelSplitComplete (a boolean that flags the final element) instead of the size. A second surprise: after .end() the original whole-order exchange resumes — not the last line item — so anything you route after the split block sees the composite body again, not SKU-3.

Production version

Here the order arrives on an in-memory direct: channel and each item lands on a log: endpoint; in production the source is usually a jms:/kafka: queue or a file: batch, and {{ep.item}} points at a real per-item handler (another jms:/kafka: topic) so items can be processed and scaled independently. For large batches add .streaming() to hold constant memory, and add .parallelProcessing() when per-item throughput matters — but know that parallelism gives up the in-order delivery this lesson relies on, so re-order with the CamelSplitIndex or an Aggregator if order is required. Give the split block its own error handling (a per-item Dead Letter Channel) so one bad line item doesn't sink the whole batch, and keep ep.item plus any parallel/streaming toggles externalized in config. Finally, wire in observability — Camel's Micrometer metrics and the named routeId("splitter") make the fan-out rate and per-item failures visible per hop.

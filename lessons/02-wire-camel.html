Wire Camel: the BOM & One-Starter-Per-Component

Add Apache Camel to a plain Spring Boot app — and prove it boots. The smallest possible first step before any pattern, plus the three facts that make Camel-on-Spring-Boot painless.

The idea

"Adding Camel" is not a framework rewrite — it is adding one dependency. Put camel-spring-boot-starter on the classpath and Spring Boot auto-creates a single CamelContext and auto-discovers every RouteBuilder you mark @Component. Two more facts round it out: the imported Camel Spring Boot BOM supplies every version, so you never write a <version>; and the rule is one starter per URI scheme — camel-core's built-in schemes come free, everything else needs its own camel-<name>-starter.

The scenario

There is no business scenario yet — this module is the wiring lesson. A heartbeat route fires from timer:hello about once a second and writes a constant body to log:hello: if you see the line, Camel started. A ping/pong route reads direct:ping, prepends "pong:", and sends it to {{ep.out}} — a property placeholder resolving to log:out when you run and mock:out in tests. Every scheme here (timer:, log:, direct:, mock:) is built into camel-core, so no extra starter is needed.

The routes

HelloTimerRoute.java · PingRoute.java
@Component
public class HelloTimerRoute extends RouteBuilder {
    public void configure() {
        from("timer:hello?period=1000")             // (1) timer: — built into camel-core, no starter
            .autoStartup("{{app.demo.enabled}}")    // (2) on for `run`, off for tests (a property)
            .routeId("hello-timer")                 //     always name a route
            .setBody(constant("Camel is wired and running ✅"))
            .to("log:hello?showBody=true");         //     log: — also built-in
    }
}

@Component
public class PingRoute extends RouteBuilder {
    public void configure() {
        from("direct:ping")                         // (3) direct: — built-in, in-memory, synchronous
            .routeId("ping")
            .transform(simple("pong:${body}"))      // (4) prepend "pong:" to whatever came in
            .to("{{ep.out}}");                      // (5) placeholder: log:out (run) / mock:out (test)
    }
}
  1. (1) timer: fires roughly once a second; it and log: are built into camel-core, so neither needs an extra -starter.
  2. (2) .autoStartup("{{app.demo.enabled}}") gates the heartbeat — true for run, false for tests — so tests never see timer noise.
  3. (3) direct: is a synchronous in-memory channel, also built-in, giving the test a deterministic entry point.
  4. (4) transform(simple("pong:${body}")) prepends "pong:" using the Simple language camel-core ships.
  5. (5) {{ep.out}} is a property placeholder: log:out when the app runs, mock:out in tests — same route, two wirings.

The test is the spec

WireCamelTest.java
@Test
void camelContextWasAutoCreatedAndRoutesDiscovered() {
    // Proof that the one starter wired Camel: the context exists and both @Component routes loaded.
    assertNotNull(context, "camel-spring-boot-starter should auto-create a CamelContext");
    assertNotNull(context.getRoute("ping"), "PingRoute should be auto-discovered");
    assertNotNull(context.getRoute("hello-timer"), "HelloTimerRoute should be auto-discovered");
}

This is the spec for the whole module: adding one starter really did create a CamelContext and auto-discover both @Component routes by ID. Two sibling tests then prove behaviour — a reply on mock:out that starts with pong:, and that it prepends the original body exactly (pong:hello). Run just the tests:

Try it Point a route at a scheme with no starter — e.g. .to("jms:queue:demo") — and watch Camel fail fast at startup with No component found with scheme: jms. The fix is never a version; it is adding camel-jms-starter, because the BOM supplies the version.

What you learned

  • Adding Camel is adding one dependency: camel-spring-boot-starter auto-creates the CamelContext and auto-discovers every @Component RouteBuilder.
  • The Camel Spring Boot BOM supplies every version, so you never write a <version> for a camel-* dependency.
  • One starter per URI scheme — camel-core's timer:/log:/direct:/mock: are free; any other scheme needs its own camel-<name>-starter or Camel fails fast at startup.

Common mistake

The silent one: forgetting @Component on a RouteBuilder. Unlike a missing starter (which fails fast with No component found with scheme:), an undiscovered route produces no error at all — the app boots, the CamelContext starts, and your route simply never exists. Always annotate the RouteBuilder with @Component (and give it a .routeId(...)) so it is auto-discovered and easy to assert on, exactly as the wiring test does.

Production version

In real life {{ep.out}} would not resolve to log:out — it would point at a durable broker such as jms:queue:orders or kafka:orders, each pulling in its own camel-jms-starter / camel-kafka-starter (still versionless, because the BOM supplies it). You would externalize those URIs and any broker credentials into application.yaml, environment variables, or a config server rather than hardcoding them, and gate autoStartup per environment via a property just as the heartbeat does here. The timer:/log: heartbeat is a teaching prop — production liveness comes from Spring Boot Actuator plus camel-micrometer metrics scraped by your monitoring stack, not a log line printed every second.

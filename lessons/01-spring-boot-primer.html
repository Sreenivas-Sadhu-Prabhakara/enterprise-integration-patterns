Spring Boot in 30 Minutes (for Camel)

Get comfortable with the Spring Boot foundations every later module relies on — before any Camel appears. Once these ideas click, the Camel modules are just "the same Spring Boot app, plus routes."

The idea

A Spring Boot app is a container of beans — plain classes Spring creates and wires for you. Put @SpringBootApplication on the entry point and it turns on component scanning, so any class annotated @Service or @Component becomes a managed bean automatically — no XML, no manual new. When one bean needs another, Spring supplies it through dependency injection instead of you constructing it. And settings live outside the code: @Value("${...}") binds a property from application.yaml straight into a field. That is the whole toolkit this course stands on.

The scenario

A one-screen app that greets someone at startup. GreetingService (a @Service) builds "<prefix>, <name>!", where the prefix comes from config. StartupRunner (a @Component that implements CommandLineRunner) runs once when the context is ready, receives the service by constructor injection, and logs greet(name) for a name also read from config. Change app.greeting.prefix or app.greeting.name in application.yaml and the output changes with no recompile. No Camel, no broker, no network — everything runs in-memory.

The code

Application · GreetingService · StartupRunner
@SpringBootApplication                                    // (1) config + auto-config + component scan of this package
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);    // (2) boot the context, then run CommandLineRunners
    }
}

@Service                                                  // (3) a scanned, singleton business bean
public class GreetingService {
    @Value("${app.greeting.prefix}")                      // (4) bound from application.yaml — no hardcoding
    private String prefix;

    public String greet(String name) {
        return prefix + ", " + name + "!";
    }
}

@Component                                                // (5) another scanned bean; CommandLineRunner runs at startup
public class StartupRunner implements CommandLineRunner {
    private final GreetingService greetingService;
    @Value("${app.greeting.name}")
    private String name;

    public StartupRunner(GreetingService greetingService) {   // (6) constructor dependency injection
        this.greetingService = greetingService;
    }

    @Override
    public void run(String... args) {
        log.info(greetingService.greet(name));
    }
}
  1. (1) @SpringBootApplication bundles three things — this class can define beans, sensible defaults are auto-configured from the classpath, and this package and below is scanned for your beans.
  2. (2) SpringApplication.run boots the ApplicationContext (the bean container): it scans, injects dependencies, binds config, then invokes every CommandLineRunner.
  3. (3) @Service marks GreetingService as a managed bean, so the container creates exactly one instance you can inject anywhere.
  4. (4) @Value binds app.greeting.prefix from application.yaml (or an env var / CLI override) into the field — change the config, change the behaviour, no recompile.
  5. (5) @Component + CommandLineRunner makes StartupRunner run once at the end of startup — the natural place to see your beans working.
  6. (6) constructor injection — Spring sees the constructor needs a GreetingService, finds the single bean of that type, and passes it in. The field is final and never new-ed by us.

Run it

terminal
./mvnw -pl patterns/01-spring-boot-primer spring-boot:run

On startup you'll see a single line — Hello, World! — logged by StartupRunner. Override the config without editing files to watch it change:

terminal
./mvnw -pl patterns/01-spring-boot-primer spring-boot:run \
  -Dspring-boot.run.arguments="--app.greeting.prefix=Namaste --app.greeting.name=Sri"
# -> Namaste, Sri!

Behind a firewall or without Red Hat access? Add -P upstream for the plain Apache Camel profile — same result, since there are no Camel deps in this module yet.

The test is the spec

GreetingServiceTest.java
@Test
void greetIncludesConfiguredPrefixAndName() {
    String result = greetingService.greet("Sri");

    assertThat(result)
        .contains(configuredPrefix)   // the @Value-injected prefix from application.yaml
        .contains("Sri");             // the name we passed in
}

@SpringBootTest boots the real ApplicationContext — the same container main() would build — proving component scan, bean creation, constructor injection and @Value binding all wire up. The test profile deliberately sets a different prefix (Namaste), and the assertion reads that injected property rather than hardcoding it, so it follows the config. Run just the tests:

terminal
./mvnw -pl patterns/01-spring-boot-primer test
Try it Add an app.greeting.punctuation property, bind it in GreetingService with a second @Value, and write the assertion that proves greet("Sri") ends with it.

What you learned

  • @SpringBootApplication + component scanning turns plain @Service / @Component classes into managed beans — no XML, no manual new.
  • Constructor dependency injection hands a bean its collaborators; the field stays final, explicit, and easy to test.
  • @Value binds application.yaml properties at runtime — the same externalised-configuration idea Camel later uses for its endpoint placeholders.

Request-Reply vs Event Message: InOut vs InOnly

Two message exchange patterns, side by side. Request-Reply (InOut) sends a message and blocks until a correlated reply comes back; Event Message (InOnly) announces that something happened and moves on. Reach for InOut when you need the answer now to keep going, InOnly when you just want to notify.

The idea

A Request-Reply is a phone call: you ask, then wait on the line for the answer before you do anything else. An Event Message is a postcard: you drop it in the box and walk away — you neither expect nor wait for a reply. In Camel the caller picks the mode at the ProducerTemplate: requestBody(...) is the InOut verb (blocks for the reply), while sendBody(...) is the one-way InOnly verb. Choosing the wrong one is a classic integration bug: waiting for a reply that never comes, or fire-and-forgetting an operation whose result you actually needed.

The scenario

ShopFlow does two things when an order lands. It authorizes the payment with a Request-Reply call on direct:authorize (route payment) and blocks for the reply body AUTH-<orderId>. Separately it announces "order received" as a one-way Event Message on direct:orders (route emit-event) that publishes to {{ep.event}} — a topic in production, a mock: endpoint in tests — and gets nothing back.

The route

RequestReplyRoute.java
from("direct:authorize")                                   // (1) the request channel
    .routeId("payment")                                    //     always name a route (tracing, metrics, tests)
    .log("Authorizing order ${body.orderId} for amount ${body.amount}")
    .setBody(simple("AUTH-${body.orderId}"));              // (2) the reply — requestBody(...) returns THIS
EventMessageRoute.java
from("direct:orders")                                      // (3) the channel order events arrive on
    .routeId("emit-event")                                 //     always name a route
    .setExchangePattern(ExchangePattern.InOnly)            // (4) fire-and-forget: no reply channel
    .log("Emitting order-received event for ${body.orderId} (no reply awaited)")
    .to("{{ep.event}}");                                   // (5) publish the event (mock: in tests, jms:/log: in prod)
  1. (1) the payment route is a responder: it consumes an order from direct:authorize.
  2. (2) it replaces the body with an auth code. A caller obtains it synchronously with requestBody("direct:authorize", order) — the InOut verb blocks until this reply comes back, correlated for you in-process.
  3. (3) the emit-event route consumes order events from direct:orders.
  4. (4) setExchangePattern(InOnly) makes the one-way intent explicit, so even an InOut caller publishes this event fire-and-forget with no reply channel.
  5. (5) the destination is a {{ep.event}} placeholder — a real topic in production, a mock: endpoint under test. With no reply channel, the emitter is decoupled from any 0..N consumers.

Run it

terminal
./mvnw -pl patterns/21-request-reply-and-event-message spring-boot:run

A demo feeder fires every 3s: it does a blocking Request-Reply call to get AUTH-A-100x, then emits a one-way Event Message — so you'll see both a correlated AUTH-… reply and an event landing in the log. Behind a firewall? Add -P upstream.

The test is the spec

RequestReplyAndEventMessageTest.java
@Test
void requestReplyReturnsCorrelatedAuthCode() {
    // InOut: requestBody blocks and returns the reply the payment route set on the body.
    String auth = template.requestBody("direct:authorize", new Order("A-1001", 999), String.class);

    assertEquals("AUTH-A-1001", auth);
}

@Test
void eventMessageIsFireAndForgetInOnly() throws Exception {
    event.expectedMessageCount(1);

    // InOnly: one event goes out; there is no reply to capture.
    template.sendBody("direct:orders", new Order("A-2002", 500));

    event.assertIsSatisfied();
    // Exactly one message, and it arrived one-way — the recorded pattern is InOnly (no reply channel).
    assertEquals(ExchangePattern.InOnly, event.getReceivedExchanges().get(0).getPattern());
}

The first test proves Request-Reply: requestBody(...) blocks and hands back the correlated AUTH-A-1001. The second proves Event Message: sendBody(...) lands exactly one message on mock:event whose recorded pattern is InOnly — fire-and-forget, no reply channel. Run just the tests:

terminal
./mvnw -pl patterns/21-request-reply-and-event-message test
Try it Change the payment route to decline orders whose amount exceeds 10,000 with a reply of DECLINED-${body.orderId}, and write the InOut assertion that proves it. Then confirm the InOnly event on direct:orders still fires exactly once, regardless of the auth outcome.

What you learned

  • Request-Reply is InOut: requestBody(...) blocks until a correlated reply comes back — use it when you need the answer to keep going.
  • Event Message is InOnly: sendBody(...) fires and forgets, and setExchangePattern(InOnly) makes that one-way intent explicit, decoupling the emitter from 0..N consumers.
  • Picking the wrong pattern is a classic bug — waiting for a reply that never comes, or fire-and-forgetting an operation whose result you needed.

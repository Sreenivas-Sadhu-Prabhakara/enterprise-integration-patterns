Aggregator: Re-Assemble Messages into One

Combine independent, related messages back into one. Reach for it whenever results computed apart — line items, shard responses, batch rows — must be re-assembled and released together once the group is complete.

The idea

Picture a delivery depot gathering every parcel bound for one address before the van leaves. An Aggregator does that for messages: it holds related messages until the group is complete, then releases a single combined message. It is the mirror image of the Splitter — where a Splitter fans one message out into many, an Aggregator collects the many back into one. In Camel it's the aggregate() block, and it needs three things: a correlation expression (which messages belong together), an AggregationStrategy (how to fold two into one), and at least one completion condition (when to let go). Omit the completion condition and the group waits forever — so aggregate() always needs one.

The scenario

ShopFlow prices and validates each line item of an order independently, then must send exactly one confirmation per order. Every item arrives on direct:items carrying two headers: orderId — the correlation key that says which order the item belongs to — and total, how many items that order will send, which drives completion. The Aggregator groups items by orderId, folds each into an OrderConfirmation, and emits the confirmation when all expected items are in, or after a short timeout if one never arrives.

The route

AggregatorRoute.java
from("direct:items")                                       // (1) each line item arrives on its own
    .routeId("aggregator")                                 //     always name a route (tracing, metrics, tests)
    .log("Item ${body.sku} for order ${header.orderId} (${header.total} expected)")
    .aggregate(header("orderId"), new LineItemAggregationStrategy()) // (2) correlate on orderId + how to merge
        .completionSize(header("total"))                   // (3) complete when all expected items are in…
        .completionTimeout(2000)                           // (4) …or after 2s of silence (the safety net)
        .log("Confirmation for order ${body.orderId}: ${body.totalAmount} total")
        .to("{{ep.out}}")                                  // (5) emit ONE aggregated confirmation
    .end();
  1. (1) each item is its own message on the in-memory direct:items channel; the orderId and total ride as headers, not in the body.
  2. (2) aggregate() opens the Aggregator: the first argument is the correlation expression (header("orderId")), the second is the LineItemAggregationStrategy that says how to fold two exchanges into one — seeding a fresh OrderConfirmation on the first item, then adding each later item to it.
  3. (3) completionSize(header("total")) releases the group the moment the expected number of items has arrived — the happy path.
  4. (4) completionTimeout(2000) is the safety net: if an expected item never shows up, the partial group is still released after 2s so it never blocks forever.
  5. (5) exactly one aggregated confirmation is emitted to the {{ep.out}} placeholder — a real direct:/jms: endpoint in production, mock:out in tests.

A demo feeder emits a fresh order every 6 seconds, splits it into 3 line items, and feeds them to the Aggregator — so you see three Item … lines followed by one Confirmation for order A-… : 100 total, the whole splitter → aggregator round-trip. Behind a firewall? Add -P upstream.

The test is the spec

AggregatorTest.java
@Test
void threeLineItemsBecomeOneConfirmation() throws Exception {
    out.expectedMessageCount(1);

    Map<String, Object> headers = headers("A-1", 3);
    template.sendBodyAndHeaders("direct:items", new LineItem("SKU-BOOK", 30), headers);
    template.sendBodyAndHeaders("direct:items", new LineItem("SKU-PEN", 5), headers);
    template.sendBodyAndHeaders("direct:items", new LineItem("SKU-BAG", 65), headers);

    // completionSize (== total header, 3) fires as soon as the third item arrives.
    out.assertIsSatisfied();

    OrderConfirmation confirmation = out.getReceivedExchanges().get(0).getIn().getBody(OrderConfirmation.class);
    assertEquals("A-1", confirmation.getOrderId());
    assertEquals(3, confirmation.getItems().size());
    assertEquals(100, confirmation.getTotalAmount());   // 30 + 5 + 65
}

In the test profile {{ep.out}} resolves to mock:out, so the test asserts exactly one confirmation is emitted, then inspects it — the right order id, all 3 items, and the folded total of 100. What you learned

  • An Aggregator is the mirror of the Splitter: it correlates related messages by a key and merges them back into a single message.
  • The AggregationStrategy holds the merge logic — oldExchange is null for the first message of a group, then each later message is folded into the running accumulator.
  • A completion condition decides when to release: completionSize for the happy path, completionTimeout as a safety net so a missing message never blocks the group forever.

Common mistake

The number-one Aggregator bug lives in the AggregationStrategy: oldExchange is null for the first message of every new group, so code that dereferences it without a guard throws a NullPointerException on item one and the group silently fails. Always branch on oldExchange == null to seed a fresh accumulator, then fold each later message into it — exactly what LineItemAggregationStrategy does. The runner-up is forgetting a completion condition altogether: with no completionSize/completionTimeout the group is held forever and nothing is ever emitted — which is why this route pairs the size rule with a timeout safety net.

Production version

The teaching route keeps in-flight groups in Camel's default in-memory aggregation repository, so a restart mid-order loses every partially-built confirmation. In production you point aggregationRepository(...) at a persistent store — JDBC, Infinispan, or a Kafka-backed repo — so pending groups survive restarts and can be recovered, and you swap {{ep.out}} from log:/mock: to a real jms: or kafka: endpoint. Externalize the completion tuning — completionSize, completionTimeout, and a completionTimeoutCheckerInterval — as config rather than hard-coded literals, and if you aggregate across a cluster enable optimisticLocking (or a shared repository) so two nodes don't fold the same correlation key concurrently. Finish with observability: the named routeId("aggregator") plus Micrometer metrics let you watch group counts, timeouts, and completion latency in real time.

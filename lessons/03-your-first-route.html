Your First Route: from(), to(), routeId()

Read any Camel route left-to-right, top-to-bottom. It begins at one from(...) consumer, flows through zero or more steps, and ends at a to(...) producer. That single shape underlies every other pattern in this course.

Reality check These direct:/seda: hops run in memory so you can focus on the pattern. Real endpoints cross a network — latency, dropped connections, duplicate deliveries. You'll make routes robust with error handling (lesson 07), real brokers (17), and the Dead Letter Channel (19).

The idea

A Camel route is a small pipeline with a fixed grammar: from( consumer ) … steps … to( producer ). The from(...) names the channel messages enter on; each step in between transforms, logs, or decides; the to(...) names where the message goes next. Master this one shape and every later pattern — routers, splitters, filters — is just more steps in the middle. And there is one habit that pays off forever: always give a route a routeId(...). Without it Camel auto-generates ids like route1, route2 from start-up order — numbers that drift the moment you add or reorder a route, quietly breaking your traces, metrics, and tests.

The scenario

ShopFlow receives orders on the direct:orders channel. Our first route, order-intake, does the smallest useful thing: it logs the order and hands it to the next stop. The producer target is a property placeholder {{ep.out}} — in production a real direct:/jms: endpoint; in tests it resolves to a mock: so we can prove exactly what the route emits.

The route

OrderIntakeRoute.java
from("direct:orders")                                   // (1) CONSUMER: the channel orders arrive on
    .routeId("order-intake")                            // (2) ALWAYS name the route
    .log("Received order ${body.orderId} from ${body.customer}") // (3) a step: one line in the flow
    .to("{{ep.out}}");                                  // (4) PRODUCER: where the message goes next
  1. (1) from("direct:orders") is the consumer — the single entry point. Every route has exactly one.
  2. (2) routeId("order-intake") pins a stable name so tracing, Micrometer metrics, and management/tests all address the route by a name that never drifts. Rule of thumb: every from(...) gets a .routeId(...) on the very next line.
  3. (3) .log(...) is a step — the simplest kind. The ${body.orderId} / ${body.customer} tokens read the order via the Simple language.
  4. (4) to("{{ep.out}}") is the producer. The {{ep.out}} placeholder lets the same route point at a real endpoint in production and a mock in tests.

A demo feeder injects a sample order every 3 seconds, so you'll see lines like Received order A-1001 from Asha as each order flows through order-intake. Notice the feeder is also a route with its own .routeId("demo-feeder") — two routes, two ids. Behind a firewall? Add -P upstream for plain Apache Camel.

The test is the spec

OrderIntakeRouteTest.java
@Test
void oneOrderInOneMessageOut() throws Exception {
    out.expectedMessageCount(1);

    template.sendBody("direct:orders", new Order("A-1001", "Asha"));

    MockEndpoint.assertIsSatisfied(context);
}

In the test profile {{ep.out}} resolves to mock:out, so we send one Order into direct:orders and assert exactly one message reaches the producer — no broker required. A second test proves the body passes through unchanged. Run just the tests:

What you learned

  • Every Camel route has the same shape: one from(...) consumer, zero or more steps, one to(...) producer.
  • routeId(...) is not optional — a stable name keeps traces, metrics, and tests pointing at the right route.
  • A property-placeholder endpoint like {{ep.out}} lets the same route be production-real and test-assertable against a mock:.

Related patterns

  • Channel & Endpoint (direct/seda) — a closer look at what those from(...) / to(...) URIs actually name.
  • Message & Exchange — what the ${body...} tokens in your .log() step are reading.
  • Pipes & Filters — the natural next step: chaining several processing steps between from and to.
  • Testing routes — turn "the test is the spec" here into the full pattern for asserting what a route emits.

Common mistake

Skipping .routeId(...). A route with no explicit id still runs, so the omission is invisible until it hurts: Camel auto-names it route1, route2 from start-up order, and those numbers shift the instant you add, remove, or reorder a route. Your traces, Micrometer tags, and any adviceWith / stop-resume test that addresses the route by id then silently point at the wrong route. Make it a reflex: every from(...) gets a .routeId(...) on the very next line.

Production version

In production the endpoints stop being in-memory: from("direct:orders") becomes a broker consumer like from("jms:queue:orders") or from("kafka:orders"), and {{ep.out}} resolves to a real jms:/http: downstream instead of log:out — keep that URI externalized in config, exactly as the placeholder already does. Concurrency is no longer a demo timer but many broker-delivered messages at once, so the route must be thread-safe and rely on the broker (and DLQ / redelivery) for durability rather than a fire-and-forget log. Replace the teaching .log() with structured logging plus a correlation id, and wire the stable routeId into Micrometer metrics and OpenTelemetry traces so order-intake is observable end to end. The demo-feeder route is deleted entirely — real orders arrive from upstream.

Message Channel & Endpoint: direct: vs seda:

Connect two routes over an in-memory Message Channel and feel the difference between a synchronous channel and an asynchronous one — the choice between direct: and seda:.

Reality check direct: and seda: are in-memory — fast to learn on, but they vanish when the JVM stops and never touch the network. Production routes cross a broker with real latency, failures, and duplicates. You'll make them robust with error handling (07), real brokers (17), the Dead Letter Channel (19), and the Idempotent Consumer (20).

The idea

A Message Channel is a named endpoint URI that carries messages between routes. A Message Endpoint is where a route attaches to that channel: from(...) is a consumer endpoint, to(...) is a producer endpoint. The channel's scheme decides its behaviour. direct: is synchronous — the message is handled on the caller's own thread, and any exception propagates straight back. seda: is asynchronous — to("seda:work") drops the message on an in-memory queue and returns immediately; a background consumer thread picks it up later. That queue is memory-only — not durable: if the JVM stops, anything still queued is lost.

The scenario

Orders arrive on the synchronous channel direct:orders. The intake route does a quick hand-off to the asynchronous channel seda:work and returns. The process route drains seda:work with two background consumers and forwards to ep.out. Because the two routes are joined by seda:, processing runs on a different thread from whoever submitted the order — the app logs both thread names so you can watch it happen.

The routes

IntakeRoute.java + ProcessRoute.java
// --- IntakeRoute: the producing side of the channel ---
from("direct:orders")                                        // (1) synchronous entry channel — runs on the caller's thread
    .routeId("intake")                                       //     always name a route (tracing, metrics, tests)
    .log("intake received '${body}' on thread ${threadName}")
    .to("seda:work");                                        // (2) async in-memory queue — hand off and return immediately

// --- ProcessRoute: the consuming side of seda:work ---
from("seda:work?concurrentConsumers=2")                      // (3) async consumer — 2 background worker threads
    .routeId("process")
    .setHeader("processingThread", simple("${threadName}"))  // (4) record which thread actually did the work
    .log("process handling '${body}' on thread ${threadName}")
    .to("{{ep.out}}");                                       // (5) placeholder terminal: log: (run) / mock: (test)
  1. (1) intake consumes from direct:orders — a synchronous channel, so this runs on whoever called it, on their own thread.
  2. (2) .to("seda:work") is the async hand-off: it enqueues the message and returns immediately, without waiting for the work to finish.
  3. (3) process consumes seda:work?concurrentConsumers=2, so the work runs on one of two background worker threads.
  4. (4) it stamps the actual worker thread name onto the processingThread header — that is exactly what the test inspects to prove the async hop.
  5. (5) {{ep.out}} is a property placeholder: a log: endpoint when you run the app, a mock: endpoint under test.

The test is the spec

MessageChannelAndEndpointTest.java
@Test
void processingRunsOnADifferentThreadThanTheCaller() throws Exception {
    out.expectedMessageCount(1);
    String callerThread = Thread.currentThread().getName();

    template.sendBody("direct:orders", "PING");

    out.assertIsSatisfied();

    // The process route stamped the thread it actually ran on. Because the hop is via seda:,
    // that is a background consumer thread, never the caller's (this JUnit) thread.
    String processingThread =
        out.getReceivedExchanges().get(0).getMessage().getHeader("processingThread", String.class);
    assertNotNull(processingThread, "process route should record the thread it ran on");
    assertNotEquals(callerThread, processingThread,
        "seda: should hand the message to a background consumer thread, not the caller's thread");
}

In the test profile ep.out resolves to mock:out, so we can assert exactly what crossed the seda:work channel — no broker required. Because seda: is asynchronous, assertIsSatisfied waits (default 10s) for the queued hop to finish before checking. Run just the tests:

What you learned

  • A Message Channel is a named endpoint URI; from(...) is a consumer endpoint and to(...) a producer endpoint.
  • direct: is synchronous (caller's thread, exceptions propagate back); seda: is asynchronous (in-memory queue, background threads, not durable).
  • Property-placeholder endpoints like {{ep.out}} let the same route run real in production and assert against mock: in tests.

Common mistake

Treating seda: like a durable broker. Because to("seda:work") returns immediately, it is tempting to lean on it for reliable, at-least-once delivery — but the queue lives only in JVM memory. If the process crashes or is redeployed, everything still sitting in seda:work is silently lost, and there is no transaction rollback to save it. Use seda: only for in-process decoupling within a single JVM; when messages must survive a restart, put a real broker channel (jms: or kafka:) between the two routes instead.

Production version

In production you would rarely join two routes over an in-memory seda: queue. The intake route would hand off to a durable channel — jms:queue:orders or kafka:orders — so a redeploy or crash never drops work, and the process route would consume that broker with the acknowledgement, redelivery, and dead-letter semantics the broker provides. Concurrency moves from concurrentConsumers to broker-level scaling (a JMS listener pool, or a Kafka consumer group spread across pods), and you would externalize the endpoint URIs, credentials, and thread counts into config or environment variables rather than hard-coding them. The {{ep.out}} placeholder already models this — it points at log: here, but in a real system it resolves to the next service endpoint. Add Micrometer metrics and distributed tracing so you can watch queue depth, consumer lag, and per-route latency instead of eyeballing thread names in the console.

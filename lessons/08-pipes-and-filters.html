Pipes and Filters: a Pipeline of Reusable Steps

Compose behaviour from a chain of small, single-responsibility steps connected by channels. Reach for it whenever a task is really several small transformations in a row.

The idea

Think of an assembly line: each station does one small job and hands the work to the next. Pipes and Filters structures a message flow the same way — independent processing steps ("filters") connected by channels ("pipes"). In Camel a plain sequence of .bean(...) / .to(...) calls is the pipeline: each step receives the previous step's output as its input. Because each filter does exactly one thing, you can reason about it, reuse it, reorder it, and unit-test it in isolation.

The scenario

An incoming ShopFlow order needs three independent bits of processing before it moves on. Rather than one big method, each is its own @Component filter: NormalizeBean trims and upper-cases the country ( in  → IN), EnrichBean stamps a receivedAt timestamp, and ValidateBean rejects the order if its orderId is missing. The output channel is a placeholder {{ep.out}} — a log: endpoint when you run the app, a mock: endpoint under test.

The route

PipesAndFiltersRoute.java
from("direct:orders")                 // (1) the channel orders arrive on
    .routeId("pipeline")              //     always name a route (tracing, metrics, tests)
    .log("Pipeline in:  ${body}")
    .bean(NormalizeBean.class)        // (2) filter: trim + upper-case the country
    .bean(EnrichBean.class)           // (3) filter: stamp a receivedAt timestamp
    .bean(ValidateBean.class)         // (4) filter: reject the order if orderId is missing
    .log("Pipeline out: ${body}")
    .to("{{ep.out}}");                // (5) the pipeline's output channel
  1. (1) the route consumes orders from an in-memory direct: channel; routeId names the pipeline for tracing and tests.
  2. (2) NormalizeBean is the first filter — it trims and upper-cases the country so every later stage can rely on a canonical IN.
  3. (3) EnrichBean stamps a receivedAt timestamp; its output becomes the next filter's input automatically.
  4. (4) ValidateBean is the last filter — it throws if orderId is missing, so a bad order never reaches the output.
  5. (5) the output target is a {{ep.out}} placeholder, so production points it at a real channel while tests point it at a mock.

Run it

terminal
./mvnw -pl patterns/08-pipes-and-filters spring-boot:run

A demo feeder injects a rotating messy sample order every 3s — watch the raw body logged at Pipeline in: (e.g. country=' in ') come out at Pipeline out: normalised to country='IN' with a receivedAt timestamp. Behind a firewall? Add -P upstream.

The test is the spec

PipesAndFiltersTest.java
@Test
void pipelineNormalisesAndEnriches() throws Exception {
    out.expectedMessageCount(1);

    // Raw input: country has surrounding spaces and is lower-cased.
    template.sendBody("direct:orders", new Order("A-1001", "  in  ", 999));

    out.assertIsSatisfied();

    Order result = out.getReceivedExchanges().get(0).getIn().getBody(Order.class);
    assertEquals("IN", result.getCountry(), "NormalizeBean should trim + upper-case the country");
    assertNotNull(result.getReceivedAt(), "EnrichBean should stamp a receivedAt timestamp");
    assertEquals("A-1001", result.getOrderId(), "orderId should flow through untouched");
}

In the test profile {{ep.out}} resolves to mock:out, so the test inspects the body the pipeline emits after all three filters have run — proving it is normalised and enriched. A companion test sends an order with no orderId and asserts it is rejected before reaching the output. Run just the tests:

terminal
./mvnw -pl patterns/08-pipes-and-filters test
Try it Add a fourth filter — a MaskBean that redacts the customer's email — as its own @Component, drop it into the chain with one .bean(MaskBean.class) call, and write a plain unit test for it in isolation (no Spring), like NormalizeBeanTest.

What you learned

  • Pipes and Filters composes a task from small single-responsibility steps ("filters") connected by channels ("pipes").
  • A chain of .bean(...) / .to(...) calls is Camel's default pipeline — each step's output is the next step's input.
  • Small filters are independently reusable, reorderable, and unit-testable, as NormalizeBeanTest proves in microseconds.

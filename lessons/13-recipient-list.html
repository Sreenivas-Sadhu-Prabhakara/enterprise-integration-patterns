Recipient List: Fan Out to a Dynamic Set

Route one message to a set of recipients that is computed at runtime. Reach for it when "who needs a copy" is decided per message — not fixed in the route.

The idea

A multicast() fans a message out to a fixed list baked into the route. A Recipient List does the same fan-out, but the destination list is data-driven: each message carries (or computes) its own set of recipients. In Camel that's recipientList(expression) — it evaluates an expression to a list of endpoint URIs and delivers a copy of the message to every one. It is the dynamic sibling of multicast().

The scenario

Every ShopFlow order must reach some systems always and others only sometimes: warehouse and invoicing get every order (it ships and it bills), but analytics is notified only for high-value orders — those whose totalAmount is at or above the configured threshold (default 1000). A RecipientResolver bean builds the list per message and returns it as a comma-separated string of endpoint URIs.

The route

RecipientListRoute.java
from("direct:orders")                                                 // (1) the channel orders arrive on
    .routeId("recipient-list")                                        //     always name a route (tracing, metrics, tests)
    .log("Fanning out order ${body.orderId} (amount ${body.totalAmount})")
    .setHeader("recipients", method(RecipientResolver.class, "recipientsFor")) // (2) compute the recipient list per message
    .recipientList(header("recipients"), ",");                        // (3) split on ',' and deliver a copy to each recipient
  1. (1) the route consumes orders from an in-memory direct: channel; routeId names it for tracing, metrics, and tests.
  2. (2) setHeader calls the RecipientResolver bean via method(...). It always appends warehouse + invoicing, and adds analytics only when totalAmount >= threshold — this per-message computation is what makes the list dynamic.
  3. (3) recipientList(header("recipients"), ",") splits that header on the comma and delivers a copy of the message to each URI. The base endpoints are {{ep.*}} placeholders, so production points them at real jms:/http: systems while tests point them at mocks. Fan-out is sequential by default; add .parallelProcessing() to dispatch concurrently.

Run it

terminal
./mvnw -pl patterns/13-recipient-list spring-boot:run

A demo feeder injects a sample order every 3s, alternating between low and high amounts — watch low-value orders land on log:warehouse and log:invoicing only, while high-value ones also hit log:analytics. Behind a firewall? Add -P upstream.

The test is the spec

RecipientListTest.java
@Test
void highValueOrderAlsoHitsAnalytics() throws Exception {
    warehouse.expectedMessageCount(1);
    invoicing.expectedMessageCount(1);
    analytics.expectedMessageCount(1);

    // 5000 is at/above the high-value threshold (1000): analytics is added to the recipient list.
    template.sendBody("direct:orders", new Order("A-2002", "IN", 5000));

    MockEndpoint.assertIsSatisfied(context);
}

In the test profile the recipient endpoints resolve to mock:warehouse, mock:invoicing, mock:analytics, so each test asserts exactly which systems an order fanned out to. Its twin, lowValueOrderSkipsAnalytics, sends an amount of 250 and proves analytics receives zero messages. Run just the tests:

terminal
./mvnw -pl patterns/13-recipient-list test
Try it Add a fourth recipient — a mock:fraud-check endpoint that only receives orders from a flagged country. Extend RecipientResolver to append it conditionally, then write the assertion that proves it fires for that country and stays empty otherwise.

What you learned

  • A Recipient List fans one message out to a set of destinations computed at runtime, so the set can differ per message.
  • recipientList(header("recipients"), ",") splits a data-driven expression and delivers a copy to each — the dynamic sibling of a static multicast().
  • Property-placeholder endpoints ({{ep.*}}) let the same route hit real systems in production and mock: endpoints under test.

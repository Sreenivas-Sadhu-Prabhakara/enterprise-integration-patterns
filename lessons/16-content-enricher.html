Content Enricher: Augment an Order with Data

Add data a message is missing by fetching it from another source and merging it in. Reach for it whenever a message carries a reference — an id, a code — but the next step needs the full record behind it.

The idea

A boarding pass shows a seat number, but the gate agent needs your whole reservation. A Content Enricher fills that gap: it takes a message holding only a reference and looks up the full record to attach. In Camel the verb is enrich() — it drives a resource route on demand (request/reply) and hands the reply to an AggregationStrategy that merges it onto the original body. Its sibling pollEnrich() instead consumes from a resource (read a file, poll a queue) — same merge idea, different trigger.

The scenario

ShopFlow orders arrive knowing only which customer placed them (customerId) — not who they are. Before an order moves downstream we enrich it with the customer's name, email and loyalty tier, looked up from an in-memory directory (a stand-in for a DB or REST service — no real HTTP). Order C-1 gains Alice Kumar / GOLD, C-2 gains Bruno Lang / SILVER, and an unknown C-9 flows on unchanged — the order is never dropped.

The route

ContentEnricherRoute.java
from("direct:orders")                                          // (1) orders arrive knowing only customerId
    .routeId("enricher")
    .log("Enriching order ${body.orderId} for customer ${body.customerId}")
    .enrich("direct:customer-lookup", new MergeStrategy())     // (2) fetch + merge customer data
    .log("Enriched order: ${body}")
    .to("{{ep.out}}");                                         // (3) enriched order continues downstream

from("direct:customer-lookup")                                 // (4) the enrich resource route
    .routeId("customer-lookup")
    .log("Looking up customer ${body.customerId}")
    .bean(CustomerRepository.class, "findById(${body.customerId})"); // (5) bean returns a Customer by id
  1. (1) the main route consumes orders from an in-memory direct: channel; each order carries a customerId and little else.
  2. (2) enrich() is the Content Enricher: it sends the order to the resource route, waits for the reply, and passes both to MergeStrategy — the AggregationStrategy that copies the customer's fields onto the order and returns the order, so the enriched message keeps flowing.
  3. (3) the output is a {{ep.out}} placeholder — production points it at a real endpoint while tests point it at mock:out.
  4. (4)(5) the resource route looks up the record: a plain bean calls CustomerRepository.findById with the order's id and returns a Customer (or null for an unknown id, which the merge leaves untouched).

The test is the spec

ContentEnricherTest.java
@Test
void orderIsEnrichedWithCustomerData() throws Exception {
    out.expectedMessageCount(1);

    template.sendBody("direct:orders", new Order("A-1001", "C-1"));

    out.assertIsSatisfied();

    Order enriched = out.getExchanges().get(0).getIn().getBody(Order.class);
    assertEquals("A-1001", enriched.getOrderId());
    assertEquals("C-1", enriched.getCustomerId());
    // These three fields were empty on the way in and filled by the Content Enricher:
    assertEquals("Alice Kumar", enriched.getCustomerName());
    assertEquals("alice@example.com", enriched.getCustomerEmail());
    assertEquals("GOLD", enriched.getCustomerTier());
}

The order goes in carrying only an id; it comes out of mock:out carrying the customer's name, email and tier — proof the message was augmented from a second source with no broker and no HTTP. Sibling tests feed a different id (C-2) to show the lookup really keys off customerId, and an unknown id (C-9) that still reaches mock:out with its customer fields null. Run just the tests:

What you learned

  • A Content Enricher augments a message with data it lacks, fetched from a separate resource keyed off a reference the message already carries.
  • enrich() drives a resource route on demand and an AggregationStrategy merges the reply back onto the original body — pollEnrich() is the consume-flavoured sibling.
  • A lookup miss doesn't drop the message: the order flows on with empty fields, so enrichment is additive, not a gate.

Related patterns

  • 15 · Message Translator — its sibling: a Translator reshapes the data already in the message, while a Content Enricher fetches missing data and adds it.
  • 12 · Aggregator — enrich() merges the resource reply with the exact same AggregationStrategy interface the Aggregator uses to combine messages.
  • 14 · Wire Tap — a useful contrast: a Wire Tap also sends to a second route but discards its reply, whereas enrich() merges that reply back into the main flow.
  • Full EIP Map — see where Content Enricher sits among the message-transformation patterns.

Common mistake

The number-one enrich() trap is omitting the AggregationStrategy: with no strategy, Camel replaces the whole body with the resource route's reply, so your original order vanishes and only the Customer survives. Always pass a strategy (here MergeStrategy) that returns the original exchange with the fetched fields copied on — not the reply. A subtler slip is assuming the strategy only runs on a hit: it also runs on a miss with a null reply, so guard for null in aggregate() instead of dereferencing it and throwing an NPE that would drop the order.

Production version

In the teaching module the resource is an in-memory Map behind direct:customer-lookup and {{ep.out}} logs to the console. In production the lookup route calls a real system — http:/a REST client, JDBC, or a jms:/kafka: request — and {{ep.out}} points at the real downstream channel, with every URI and credential externalized to config. Because enrich() is a synchronous call to that external source, wrap it in a timeout and a circuit breaker, and cache hot lookups (Caffeine/Redis) so you don't hit the customer service on every message. Decide the failure policy deliberately: route lookup errors to a Dead Letter Channel rather than letting a slow or down dependency stall the pipeline, and add metrics/tracing on the enrich step so you can see lookup latency and miss rates.

Wire Tap: Copy Every Order to an Audit Trail

Send a copy of every message to a secondary channel without disturbing the main flow. Reach for it whenever you need to observe, log, or audit traffic and the observer must not change or slow down what the main route does.

The idea

Picture a phone line with a listening device spliced onto it: the call goes through untouched while a copy is recorded elsewhere. A Wire Tap does the same for messages — it drops a duplicate onto a secondary channel while the original keeps flowing. In Camel that's wireTap(uri). It is asynchronous, fire-and-forget: the main route continues immediately without waiting on the tap, and the copy never alters the main exchange. Contrast this with Multicast / Recipient List, which fan a message out as part of the main route and, by default, wait for every destination.

The scenario

ShopFlow must keep an audit copy of every order for compliance, but auditing must never hold up fulfilment. Each order continues down the main pipeline to ep.main, and a fire-and-forget copy is tapped onto ep.audit. Both are property placeholders — in production they point at real direct:/jms: endpoints, and in tests at mock: endpoints so we can prove the copy happened.

The route

WireTapRoute.java
from("direct:orders")                                     // (1) the channel orders arrive on
    .routeId("wire-tap")                                  //     always name a route (tracing, metrics, tests)
    .log("Order ${body.orderId} — tapping a copy to audit, main flow continues")
    .wireTap("{{ep.audit}}")                              // (2) async fire-and-forget COPY to the audit trail
    .to("{{ep.main}}");                                   // (3) the main flow continues, untouched
  1. (1) the route consumes orders from an in-memory direct: channel.
  2. (2) wireTap() drops a copy of the exchange onto the audit channel and returns immediately — it does not wait for the copy to be processed, and it does not touch the message travelling on the main route.
  3. (3) execution flows straight on to {{ep.main}}, the main destination, exactly as if the tap were not there. The two targets are {{ep.*}} placeholders so production points them at real channels while tests point them at mocks.

Run it

terminal
./mvnw -pl patterns/14-wire-tap spring-boot:run

A demo feeder injects a sample order every three seconds — watch the same order land on both log:main (main flow) and log:audit (the tapped copy). Behind a firewall? Add -P upstream.

The test is the spec

WireTapTest.java
@Test
void everyOrderIsCopiedToAudit() throws Exception {
    Order order = new Order("A-1001", "IN", 999);

    // Both the main flow and the audit trail must see exactly this one order.
    main.expectedMessageCount(1);
    main.expectedBodiesReceived(order);
    audit.expectedMessageCount(1);
    audit.expectedBodiesReceived(order);

    template.sendBody("direct:orders", order);

    // Waits for the async tap; passes only if BOTH mocks received the order.
    MockEndpoint.assertIsSatisfied(context);
}

In the test profile the two endpoints resolve to mock:main and mock:audit, so the test asserts that the order reaches the main flow and that an identical copy reaches the audit trail. Because the tap is asynchronous, MockEndpoint waits up to its timeout for the expected copy — so assertIsSatisfied handles the fire-and-forget transparently. Run just the tests:

terminal
./mvnw -pl patterns/14-wire-tap test
Try it Add a second tap that also copies each order to a mock:metrics endpoint, and write the assertion that proves all three endpoints receive every order.

What you learned

  • A Wire Tap sends a copy of every message to a secondary channel without disturbing the main flow.
  • wireTap() is asynchronous and fire-and-forget — the main route continues immediately and the tapped copy never alters the main exchange.
  • Property-placeholder endpoints let the same route be production-real and test-assertable — and a waiting MockEndpoint makes the async copy easy to assert.

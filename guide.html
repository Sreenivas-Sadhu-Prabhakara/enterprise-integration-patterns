What are Enterprise Integration Patterns?

Enterprise Integration Patterns (EIPs) are named, reusable solutions to the problems that show up whenever two systems talk to each other through messages. How do you send an order to the right warehouse? Break a bulk file into line items? Combine replies back into one result? Not lose a message when a downstream service is down? Each of those has a well-understood pattern — a Content-Based Router, a Splitter, an Aggregator, a Dead Letter Channel.

The catalog was codified by Gregor Hohpe and Bobby Woolf in their 2003 book, and it's still the shared vocabulary of integration work. This guide teaches the most useful patterns as working code. See the full EIP map for how the whole catalog maps to Camel.

Why Apache Camel and Spring Boot?

Most EIP material is either an abstract catalog or a wall of prose. Apache Camel is different: it implements the patterns directly as a small routing language. A Content-Based Router really is choice().when(...).otherwise(); a Splitter really is split(...). With 300+ connectors and built-in error handling, Camel is what a huge amount of real-world integration actually runs on.

On Spring Boot, a single starter auto-configures a CamelContext and discovers your routes, so you can run a pattern in minutes with no ceremony. This course uses the supported Red Hat build of Apache Camel for Spring Boot (Camel 4.18, Spring Boot 3.5, JDK 17/21), with a plain Apache Camel fallback so it works anywhere. New to comparing options? See Apache Camel vs Spring Integration.

The pattern families

The 21 lessons build one running example — ShopFlow, an online store's order pipeline — grouped into families:

Foundations

Start here if Spring Boot or Camel is new. Spring Boot in 30 minutes · Wire Camel · Your first route · Channels & endpoints (direct vs seda) · The message & exchange · Testing routes · Error handling basics.

Message routing

Get messages to the right place. Pipes & Filters · Message Filter · Content-Based Router · Splitter · Aggregator · Recipient List · Wire Tap.

Transformation & messaging channels

Change shape and cross real brokers. Message Translator (JSON ↔ POJO) · Content Enricher · Point-to-Point vs Publish-Subscribe.

Reliability & the capstone

Don't lose messages, and don't do work twice. Dead Letter Channel & Redelivery · Idempotent Consumer · Request-Reply vs Event Message · the end-to-end capstone that composes eight patterns.

How the course works

  • Runnable — every pattern is its own Maven module you clone and run.
  • Tested — each ships JUnit 5 tests with mock endpoints; the test is the specification of the pattern.
  • Explained — a message-flow diagram, an annotated route, and a plain-English "when to reach for this".
  • Verified — CI builds and tests every module on JDK 17 & 21, against both the Red Hat and upstream Camel builds.

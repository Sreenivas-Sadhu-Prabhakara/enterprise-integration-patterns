Pattern comparison

Request-Reply vs Event Message

Two ways to send a message: ask a question and wait for the answer, or announce a fact and move on. One line of Camel decides which — and the choice shapes your coupling, latency, and how the system fails.

The two styles in one paragraph

Request-Reply is an InOut exchange: the sender transmits a message and blocks until a correlated reply comes back. That creates temporal coupling — the caller depends on the responder being up and answering now. Reach for it when you need a result to keep going: a query, or a command whose outcome you must have, such as a payment authorization. In Camel it is ExchangePattern.InOut, and the caller uses template.requestBody(...).

Event Message is an InOnly exchange: a fire-and-forget notification that something happened, with no reply expected. That gives you loose coupling — the emitter neither knows nor waits for whoever consumes the fact. Reach for it to broadcast facts like order-placed, typically over a Publish-Subscribe channel so 0..N subscribers react independently. In Camel it is the default InOnly — an ordinary to(...), or an explicit setExchangePattern(ExchangePattern.InOnly).

Trade-offs at a glance

ConcernRequest-ReplyEvent Message
Exchange patternInOut — message out, correlated reply backInOnly — one-way, no reply channel
CouplingTemporal & behavioural: caller bound to the responder being available and answeringLoose: emitter decoupled from 0..N consumers, who may not even exist yet
LatencyCaller blocks until the reply arrives (or the call times out)Caller returns immediately; consumers process out of band
Failure handlingErrors surface to the caller synchronously — an exception on requestBody you can catch and react toNo caller to surface to — needs a Dead Letter Channel and monitoring to catch failures
ScalabilityBounded by the slowest responder; fanning out to many handlers is awkwardFans out naturally via pub/sub — add subscribers without touching the emitter
Typical useQueries and commands that need a result now: payment authorization, price lookup, validation"Something happened" facts consumed independently: order-placed, user-registered, file-uploaded

How it looks in Camel

Request-Reply (InOut). The responder route puts the answer on the body; the caller blocks for it with requestBody:

RequestReplyRoute.java
// Responder: consume the order, replace the body with the reply.
from("direct:authorize")
    .routeId("payment")
    .log("Authorizing order ${body.orderId} for ${body.amount}")
    .setBody(simple("AUTH-${body.orderId}"));      // this becomes the reply

// Caller: InOut — block until the correlated reply comes back.
String auth = template.requestBody("direct:authorize", order, String.class);
// auth == "AUTH-A-1001"

Over an in-memory direct: endpoint Camel correlates the reply for you in-process. Over an async transport such as JMS you need a reply channel and a Correlation Identifier to match answers to requests — Camel manages a temporary reply queue and the JMSCorrelationID automatically when the exchange is InOut:

JmsRequestReply.java
from("direct:pay")
    // InOut over JMS: Camel creates a temporary reply queue and correlates
    // the response by JMSCorrelationID — no manual plumbing.
    .to(ExchangePattern.InOut, "jms:queue:authorize");

Event Message (InOnly). The emitter publishes a fact and moves on. to(...) is InOnly by default; make the intent explicit so even an InOut caller fires it one-way:

EventMessageRoute.java
// Emitter: announce the fact one-way, no reply awaited.
from("direct:orders")
    .routeId("emit-event")
    .setExchangePattern(ExchangePattern.InOnly)    // explicit fire-and-forget
    .log("Emitting order-received for ${body.orderId} (no reply)")
    .to("jms:topic:order-received");               // pub/sub fan-out

// Caller: sendBody is the one-way verb — returns immediately.
template.sendBody("direct:orders", order);

Publishing to a topic gives you Publish-Subscribe: every subscriber gets its own copy, and you add or remove them without the emitter ever knowing.

Pitfalls

  • Waiting for a reply that never comes. An InOut call over a transport with no responder — or a responder that never sets a reply — blocks until it times out. Always set a sensible requestTimeout on async request-reply and handle the timeout.
  • Fire-and-forgetting a result you needed. Using sendBody (InOnly) when you actually depend on the outcome silently drops the answer. If you need the auth code, use requestBody.
  • Silent event failures. With no caller to raise the error to, a failed Event Message vanishes unless you wire a Dead Letter Channel and monitor it. "No reply" is not "no failures".
  • Request-Reply for broadcasts. If several handlers should react to the same fact, don't turn it into N blocking calls — publish one event and let pub/sub fan it out.
  • Distributed coupling. A chain of synchronous InOut hops multiplies latency and turns one slow service into a system-wide stall. Prefer events at the seams where you can tolerate eventual processing.

Rule of thumb

Verdict If the sender cannot proceed without the answer, use Request-Reply (InOut, requestBody) and accept the temporal coupling — but bound it with a timeout and handle the error. If you are merely announcing that something happened and others may or may not care, use an Event Message (InOnly, sendBody over pub/sub) and lean on a Dead Letter Channel plus monitoring for reliability. When in doubt, prefer events at service boundaries and keep Request-Reply inside a service where the round-trip is cheap.

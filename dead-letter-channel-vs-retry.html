Dead Letter Channel vs Retry (Redelivery)

A recurring question when a route can fail: do I retry, or do I dead-letter? The honest answer is both — they solve different halves of the same problem and are designed to be used together, not chosen between.

The short answer

Retry / redelivery is optimistic: it assumes the failure is transient — a blip, a timeout, a momentarily overloaded dependency — and re-attempts the same message a few times, ideally with growing delay, on the bet that it will succeed on a later try. A Dead Letter Channel (DLC) is pessimistic: it is where a message goes after retries are exhausted — or immediately, for a message that is provably unrecoverable — so it is parked for inspection and replay instead of being silently lost or crashing the route. Retry is the recovery attempt; the DLC is the safety net under it.

RETRY · REDELIVERY

Recover from transient failures

Re-attempt the same exchange a bounded number of times with backoff. Configured with a redelivery policy: maximumRedeliveries, redeliveryDelay, backOffMultiplier — on errorHandler(...) or a specific onException(...).

DEAD LETTER CHANNEL

Park what retry can't fix

Route the give-up (or poison) message to a dedicated endpoint for humans or a replay job. Configured with errorHandler(deadLetterChannel(uri)) — the redelivery policy lives on the same handler.

Transient vs. permanent — retry only what can succeed

Retrying is only useful when a later attempt might behave differently. A transient error (network timeout, HTTP 503, a locked row) fits that bill. A permanent error does not: a malformed payload, a validation failure, or a hard 400 Bad Request will fail identically on every attempt. Retrying it just delays the inevitable while hammering the dependency and holding up the exchange. Detect permanent failures and send them straight to an Invalid Message Channel (a DLC used specifically for bad data) with maximumRedeliveries(0):

OrderRoute.java — gate permanent errors
// Permanent, unrecoverable failure: never retry — divert immediately.
onException(ValidationException.class)
    .maximumRedeliveries(0)                 // (1) zero retries: a bad payload never becomes good
    .handled(true)                          //     don't rethrow to the caller
    .log("Invalid order ${body.orderId}: ${exception.message} -> invalid channel")
    .to("{{ep.invalid}}");                  // (2) Invalid Message Channel (a purpose-built DLC)

// Transient failure: the default policy below still retries these a few times.
onException(java.net.ConnectException.class)
    .maximumRedeliveries(4)                 // (3) a flaky socket is worth re-attempting
    .redeliveryDelay(200)
    .backOffMultiplier(2);                  //     200ms -> 400ms -> 800ms -> 1600ms
  1. (1) the most specific onException wins, so a ValidationException skips retries entirely — maximumRedeliveries(0) means one attempt, then divert.
  2. (2) a bad message lands on its own channel where it can be inspected and corrected, never mixed in with transient failures.
  3. (3) a ConnectException is genuinely transient, so it gets the retry treatment — same route, opposite policy.

Use them together

The everyday configuration is a single pipeline: retry N times for the transient case, and when the message still won't go through, fall through to the Dead Letter Channel. One errorHandler expresses both — the redelivery policy and the dead-letter endpoint live on the same handler.

OrderRoute.java — retry, then dead-letter
from("direct:orders")
    .routeId("orders")
    // One handler carries BOTH: the redelivery policy AND the give-up destination.
    .errorHandler(deadLetterChannel("{{ep.dead}}")   // (1) where an exhausted message is parked
        .maximumRedeliveries(3)                      // (2) retry up to 3 times for transient faults
        .redeliveryDelay(200)                        // (3) 200 ms before the first retry
        .backOffMultiplier(2)                        // (4) grow the wait: 200 -> 400 -> 800 ms
        .retryAttemptedLogLevel(LoggingLevel.WARN))  //     log each retry so a flaky dep is visible
    // Retries re-run this step, so guard side effects with an Idempotent Consumer.
    .idempotentConsumer(
            header("orderId"),                       // (5) stable business key = "same message"
            MemoryIdempotentRepository.memoryIdempotentRepository())
        .bean("paymentGateway", "charge")            // (6) the fallible call
        .to("{{ep.ok}}")                             //     success channel
    .end();
  1. (1) deadLetterChannel("{{ep.dead}}") is the terminal safety net — reached only when retries can't save the message. Because the DLC handles the exhausted exchange, the caller never sees the exception.
  2. (2)–(4) the redelivery policy is the retry half: three attempts, starting 200 ms apart and doubling, so a struggling dependency gets breathing room instead of a thundering herd.
  3. (5)–(6) because a retry re-executes charge, an idempotentConsumer keyed on orderId makes the charge happen at most once even across redeliveries and replays.

Design considerations

  • Retries duplicate side effects. A re-attempted charge, send email, or insert can happen more than once. Pair retry with an Idempotent Consumer (or make the downstream call naturally idempotent) so at-least-once delivery doesn't become at-least-twice execution.
  • Don't retry poison messages. A message that fails deterministically will burn every redelivery and then dead-letter anyway. Classify it as permanent (see above) and route it straight to the Invalid Message Channel with maximumRedeliveries(0).
  • Use backoff. Fixed, immediate retries hammer an already-failing dependency. backOffMultiplier (and optionally a maximumRedeliveryDelay ceiling) backs off gracefully and gives the dependency time to recover.
  • Alert on DLQ depth. A Dead Letter Channel only helps if someone looks. Monitor the depth of the dead-letter endpoint and page when it grows — a rising DLQ is your earliest signal that a dependency is broken.
  • Provide replay tooling. Parked messages need a path back in. Ship a way to inspect, fix, and re-inject dead letters (a route, a script, or a broker console) so the DLC is a pause, not a graveyard.
Pair them The single most common production mistake is enabling retries without an Idempotent Consumer. If a step has side effects, add de-duplication before you add redeliveries — otherwise every retry risks a double charge.

Verdict

These are not alternatives. Retry / redelivery is the recovery mechanism; the Dead Letter Channel is the terminal safety net. Configure a bounded, backing-off redelivery policy for transient failures, let the DLC catch whatever survives it, gate permanent errors to maximumRedeliveries(0) so they skip retries and go straight to an Invalid Message Channel, and make any step with side effects idempotent. Do all four and a failing message is never lost, never duplicated, and never silently stuck.
